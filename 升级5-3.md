## 现在怎么解决

### 1. 先把 Upgrade 5 改写成“迁移清单”，而不是“收尾清单”

当前 closure 分支相对 `main` 是 **1 个 commit、61 个文件变更**，提交名就是“finalize upgrade5 workflow, session, and ui integration”，而 compare 页面还提示这个 diff 太大、无法正常渲染。这个形态很适合做“旧架构收尾”，不适合做“LangGraph / Deep Agents 迁移”。所以第一步不是再追着 Codex 修细节，而是把 Upgrade 5 改成一个分阶段迁移计划。([GitHub][3])

你应该把 Upgrade 5 明确拆成这四段：

* **U5-1：Runtime 接入**

  * 引入 `langgraph`、`langchain`、`deepagents`
  * 建立统一 state schema、checkpoint、memory/store、HITL runtime
* **U5-2：三工作流 graph 化**

  * Workflow 1、2、3 从 `SupportIntakeWorkflow` / `CaseCollabWorkflow` 迁成 graph nodes + edges
* **U5-3：四 Agent Deep Agent 化**

  * Intake、Case Copilot、Operator/Supervisor、Dispatch/Collaboration 不是“接口名上的 agent”，而是实际使用 graph runtime / deep agent harness
* **U5-4：前端与渠道闭环**

  * Web Console、WeCom 跨群协同、审批、trace、session end 全部接到新 runtime

这一步的目的，是让 Codex 不再“按旧架构收尾”，而是被迫沿着你定义的升级 5 系列目标推进。这个重写是必要的，因为当前仓库自述和依赖声明都还停留在旧模式。([GitHub][1])

### 2. 先补“框架骨架”，不要先补业务枝节

你现在最容易踩的坑，是继续先修“前端为什么看不到”“WeCom 为什么没推群”“operator copilot 为什么报错”。这些当然要修，但在你坚持保留 LangGraph / Deep Agents 目标的前提下，**第一优先级应该是 runtime skeleton**。

先做这 5 件事：

1. **在后端引入 LangGraph runtime**
   把每个 ticket/session 变成 graph state，而不是继续靠散落在 `workflows/*.py` 的自定义流程函数。LangGraph 的强项就是 durable execution、stateful agents、HITL，这正好对应你这个工单平台的 ticket 生命周期、会话恢复、审批与人工接入。([LangChain 文档][2])

2. **建立统一 state model**
   至少包含：

   * `ticket`
   * `session`
   * `handoff`
   * `approval`
   * `grounding`
   * `trace`
   * `copilot_outputs`
   * `channel_route`

   这样 Workflow 1/2/3 才能共享同一个 graph state，而不是像现在这样“文档里共享 ticket core，代码里仍是分散流程”。README 虽然把三工作流和四 Agent 都定义出来了，但当前实现仍是 workflow-first。([GitHub][1])

3. **把工具层抽成 LangChain tools**
   现在 README 里列出的 `Retriever`、`SummaryEngine`、`TicketAPI`、`RecommendedActionsEngine`、队列汇总、审批等，本质上都应该变成 tools，被 graph nodes 或 deep agents 调用，而不是散落在旧 workflow 中。README 里四个 Agent 的工具边界已经写得很像“将来应该 tool 化”的样子，这正好可以直接拿来做拆分依据。([GitHub][1])

4. **把 Deep Agents 只放在复杂推理面**
   Deep Agents 适合复杂、多步骤、需要规划/上下文管理/子 agent 的任务，不适合取代所有高约束动作。也就是说：

   * `resolve / customer-confirm / operator-close / approve` 这类终态动作，仍然由 graph + policy gate 控制
   * 调查、运营问答、分派建议、复杂追问恢复，交给 deep agents
     这和 Deep Agents 官方定位是一致的：它适合 planning、subagents、memory，而不是把所有工单生命周期直接交给 agent 自治。([LangChain 文档][4])

5. **checkpoint / memory / HITL 一开始就设计进去**
   LangGraph 官方把 durable execution、memory、human-in-the-loop 作为核心能力；Deep Agents 也明确依赖 LangGraph runtime 来实现持久执行与人工参与。你这个平台的“人工接入”“审批恢复”“会话结束再继续”本来就是这类能力的天然应用场景，所以不能后补。([LangChain 文档][2])

### 3. 三工作流要怎么按 Upgrade 5 迁

这部分才是“按你升级 5 系列去完成”的核心。

#### Workflow 1：员工请求入口

现在 README 里，Workflow 1 仍是 `SupportIntakeWorkflow` 驱动，目标是把入口消息转成 FAQ 直答、建单回执、handoff decision。这个工作流最适合先 graph 化，因为它天然是“意图判断 -> 检索 -> FAQ/建单分流 -> 协同推送 -> 用户回复”的状态流。([GitHub][1])

落地时应当这样拆：

* `ingest_message`
* `classify_intent`
* `retrieve_context`
* `faq_answer_or_ticket_open`
* `session_control_detect`
* `customer_confirm_detect`
* `emit_user_reply`
* `emit_collab_push`

其中 `classify_intent`、`retrieve_context`、`customer_confirm_detect` 可以是 LangChain tool + LLM 节点；`faq_answer_or_ticket_open`、`emit_collab_push` 是 policy node；`session_control_detect` 要直接连到 session state。

#### Workflow 2：处理人员协同

现在 README 把它写成 `/claim /reassign /escalate /resolve /close /state` 命令驱动的协同工作流。这个流不应该被 Deep Agents 接管，而应该变成 **graph + command gate**。因为它是高约束流程，README 自己也强调高风险动作必须可控、可审计。([GitHub][1])

正确做法是：

* slash 命令进来先进入 `command_parse`
* 再进 `policy_gate`
* 需要审批则走 `approval_pending`
* 不需要审批才进入 `apply_ticket_transition`
* 最后触发 `notify_user / notify_collab / update_console`

这样 Upgrade 5 既保留 LangGraph 目标，也不会把终态动作交给 Deep Agents 乱跑。

#### Workflow 3：前端工作台

README 里 Workflow 3 已经定义成前端工作台处理；前端也已经在详情页里做了 `AI 助手区 / 人工动作区 / 审批恢复区` 这套骨架。问题不是“前端没做”，而是它现在只是旧 API 的前端壳，还不是新 runtime 的可视化入口。([GitHub][1])

Upgrade 5 下，Workflow 3 应该做成：

* 详情页展示 graph state
* 显示当前 node / pending approval / pending customer / pending handoff
* AI 助手区直接查询 ticket graph + deep agent outputs
* 人工动作区不是散按钮，而是 graph transition controls
* trace 页不是旧链路 replay，而是 LangGraph/LangSmith 风格的 state/edge execution trace

### 4. 四个 Agent 要怎么按 Upgrade 5 变成真 Agent

当前 README 虽然写了四个 Agent，但更多是“能力命名”，不是 framework-level agents。你要保留 LangGraph / Deep Agents 目标，就要把这四个真的落实成 runtime 里的 agent roles。([GitHub][1])

建议这样分：

#### Intake Agent

保留轻量，不用 Deep Agents。
职责是：理解入口、检索上下文、生成 ticket/open/faq 建议。
实现方式：LangGraph node + LangChain tools。

#### Case Copilot Agent

做成 ticket-scoped deep agent。
职责是：针对单 ticket 做多步调查、相似案例、grounding、追问建议。
因为这类任务需要规划和上下文收敛，适合 Deep Agents。([LangChain 文档][5])

#### Operator / Supervisor Agent

做成 queue-scoped deep agent。
职责是：队列压力、SLA 风险、优先级建议、批量运营问答。
这个 agent 很适合用 Deep Agents 做规划型多步问答，而不是仅靠“规则化回答 + grounding”。README 里现在对它的定义还停留在规则回答层，这正是 Upgrade 5 要升级的地方。([GitHub][1])

#### Dispatch / Collaboration Agent

做成 dispatch-scoped deep agent + graph gate。
职责是：分派建议、协同问答、跨组调度；
但真正的 `reassign / operator-close / approve` 仍必须经过 graph/policy gate。
这就是 Deep Agents 和 LangGraph 在你的系统里的正确分工。

* Web Console 同步展示这次 dispatch edge 和 delivery status

#### 断点三：v2 合约还没 freeze

`API_CONTRACT_V2.md` 还写着 `Status: draft`、`Phase: upgrade5-s2`。你要按 Upgrade 5 系列完成，而不是降级目标，那就必须先把这个 contract 改成“迁移中的目标 contract”，而不是继续停留 draft。([GitHub][8])

## 使用建议

你可以这么发：

先发上面的**总前置说明模板**，等 Codex 明确复述理解后，再单独发某一个任务单。
每完成一个任务，就把对应的 **验收 checklist** 原样贴过去，让它自检一遍，再由你做最终判断。

这样做的好处是两点：
第一，能把 Codex 从“继续沿旧 workflow 收尾”的惯性里拉回来，因为当前仓库公开文档本来就还是 workflow-first。第二，能强制它把交付落在 LangGraph 的 durable execution / human-in-the-loop / memory，以及 Deep Agents 的 planning / context management / subagents / long-term memory 这些真正的升级目标上，而不是只补页面或接口名。([GitHub][1])
