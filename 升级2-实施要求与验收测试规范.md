## 1. 文档目的

本文件是 `升级2.md` 的执行约束与验收补充，定义三件事：

1. 升级2必须做到什么（可执行要求）。
2. 升级2做到什么才算通过（验收标准）。
3. 升级2如何测试并形成可复核证据（测试方案）。

适用范围：`support-agent-platform` 仓库内升级2相关前后端建设与验收。

## 2. 基线与范围

### 2.1 当前基线

1. 当前系统定位是 `workflow-first, agent-assisted`。
2. OpenClaw 仅承担 `ingress/session/routing`，不承载业务规则。
3. 当前仓库以后端工作流和脚本为主，尚未形成升级2要求的完整 `/api/*` 工作台接口层。

### 2.2 升级2目标范围

升级2目标是交付“客服/运营工作台（Ops Console）”，不是用户聊天页。范围包含：

1. Dashboard。
2. Ticket List。
3. Ticket Detail。
4. Event Timeline（可嵌入 Ticket Detail）。
5. Trace List / Trace Detail。
6. Queue 页面。
7. KB 管理页面。
8. Channels / Gateway 页面。

## 3. 强制约束（MUST）

### 3.1 架构边界

1. 不得重构或削弱现有 `workflow-first` 主骨架。
2. 不得把 OpenClaw 改造成业务引擎。
3. 业务规则必须继续由 `workflows/` 与 `core/` 承载。

### 3.2 功能边界

1. 前端必须是 Ticket/Case/Object-Centered 工作台，不得退化为普通聊天 UI。
2. 升级2必须先完成“页面骨架 + API 对接”，再做主题与视觉深度定制。
3. 公司定制字段必须在列表筛选、详情展示、summary/recommended actions 中可用。

### 3.3 交付边界

1. 不允许无测试交付。
2. 不允许只交页面静态稿而无联调。
3. 不允许只补 API 而无页面验证。

## 4. 交付物要求

升级2完成时，至少应包含以下交付物：

1. `web_console/` 前端工程（含路由、组件、API client、状态处理）。
2. 工作台所需后端 API 层（`/api/*`）及统一错误返回规范。
3. 页面/API 对接说明文档（可用 `升级2.md` + 本文档）。
4. 自动化测试与验收产物（测试报告、acceptance 报告、关键截图/录屏）。

## 5. 分阶段实施与退出标准

### 5.1 第一轮（基础可用）

实现内容：

1. `web_console/` 初始化。
2. App Shell（Sidebar/Topbar/Breadcrumb）。
3. Ticket List、Ticket Detail 基础版。
4. API client。
5. 假数据联调。

退出标准：

1. 页面路由可访问且无阻断性报错。
2. Ticket List 支持最小筛选与分页展示。
3. Ticket Detail 可展示核心字段与基础动作区。

### 5.2 第二轮（主流程闭环）

实现内容：

1. Event Timeline。
2. Trace List / Trace Detail。
3. Queue Summary。
4. claim/reassign/escalate/resolve/close 动作面板。

退出标准：

1. Ticket 处理动作能从页面触发并反映状态变化。
2. Timeline 可展示关键流转事件。
3. Trace 页面可完成列表到详情 drill-down。

### 5.3 第三轮（管理与运营增强）

实现内容：

1. KB 管理（FAQ/SOP/History Case）。
2. Gateway/Channels 页面。
3. SLA 可视化。
4. 定制字段完善。

退出标准：

1. KB 支持基础 CRUD 和筛选。
2. Gateway/Channel 状态可视化可用于运维观察。
3. SLA 风险状态在页面可见且颜色语义统一。

### 5.4 第四轮（发布前打磨）

实现内容：

1. 主题样式深度定制。
2. 细粒度 Loading/Empty/Error。
3. 权限与审批入口。
4. acceptance demo polish。

退出标准：

1. 关键页面具备统一交互细节与状态反馈。
2. 演示脚本可稳定复现主链路。
3. 验收项通过率达到本文第 8 章要求。

## 6. 功能要求明细

## 6.1 页面级要求

### Dashboard

1. 必须展示核心指标：新建、处理中、SLA 预警、升级、handoff、渠道健康、trace 异常、队列负载。
2. 必须支持从异常项跳转到对应详情页。

### Ticket List

1. 必须支持筛选：`status/priority/queue/assignee/channel/handoff_state/service_type/risk_level/created_at range/sla_state`。
2. 筛选条件必须持久化（QueryString 或 LocalStorage）。
3. 表格必须支持排序、分页，导出能力可标记为后续增强。

### Ticket Detail

1. 必须展示左中右三栏核心信息、处理视图、执行状态。
2. 必须包含：summary、recommended actions、similar cases、SLA、handoff、trace 摘要。
3. 必须可执行：claim/reassign/escalate/resolve/close。

### Event Timeline

1. 必须展示关键事件节点（ticket_created 到 closed）。
2. 必须支持 hover 查看事件摘要。

### Trace List / Trace Detail

1. Trace List 必须支持 `workflow/channel/provider/error_only/handoff` 筛选。
2. Trace Detail 必须展示 route decision、retrieved docs、tool calls、summary、handoff reason、latency。

### Queue

1. 必须展示队列维度的待处理、处理中、SLA 风险、升级数量。
2. 必须支持按队列跳转 Ticket List 过滤视图。

### KB 管理

1. 必须支持 `faq/sop/history_case` 三类数据浏览。
2. 必须支持新增、编辑、删除。

### Channels / Gateway

1. 必须展示 channel connected、last event、last error、retry state。
2. 必须展示 gateway status 与 route 摘要。

## 6.2 API 要求

以下接口为升级2必须可用（至少完成最小可用字段与状态码）：

1. Dashboard：`GET /api/dashboard/summary`、`GET /api/dashboard/recent-errors`。
2. Tickets：`GET /api/tickets`、`GET /api/tickets/:ticketId`、`GET /api/tickets/:ticketId/events`、`GET /api/tickets/:ticketId/assist`、`GET /api/tickets/:ticketId/similar-cases`。
3. Ticket Actions：`POST /api/tickets/:ticketId/claim`、`/reassign`、`/escalate`、`/resolve`、`/close`。
4. Queue：`GET /api/queues`、`GET /api/queues/summary`。
5. Trace：`GET /api/traces`、`GET /api/traces/:traceId`。
6. KB：`GET /api/kb`、`POST /api/kb`、`PATCH /api/kb/:docId`、`DELETE /api/kb/:docId`。
7. Channel/OpenClaw：`GET /api/channels/health`、`GET /api/channels/events`、`GET /api/openclaw/status`、`GET /api/openclaw/routes`。
8. 支撑接口：`GET /api/agents/assignees`（用于 Ticket List/Detail 选择器）。

接口规范要求：

1. 列表接口必须支持 `page/page_size` 与总数返回。
2. 失败响应必须统一结构，至少包含 `code/message/request_id`。
3. 动作类接口必须记录审计事件并可在 Timeline 回溯。

## 6.3 公司定制字段要求

必须预留并贯通以下字段：

1. `service_type`
2. `community_name`
3. `building`
4. `unit`
5. `parking_lot`
6. `parking_space_id`
7. `rent_contract_id`
8. `complaint_category`
9. `visit_required`
10. `follow_up_required`
11. `approval_required`
12. `risk_level`

字段落地最低标准：

1. 列表可筛选。
2. 详情可展示。
3. summary 可引用。
4. recommended actions 可引用。

## 7. 交互与体验要求（MUST/SHOULD）

1. MUST：列表页统一具备 Loading、Empty、Error 三态。
2. MUST：SLA 状态颜色语义统一（normal/warning/breached）。
3. MUST：关键动作按钮具备确认与反馈（成功/失败 toast 或内联提示）。
4. SHOULD：表格列可配置显示，导出能力可延后。
5. SHOULD：页面在桌面与移动视口可正常加载，不出现结构性遮挡。

## 8. 验收标准

## 8.1 功能验收

1. 8 个目标页面均可访问并完成 API 联调。
2. Ticket 生命周期动作链可闭环：`claim -> reassign/escalate -> resolve -> close`。
3. 可从 Ticket 详情查看完整事件时间线与相关 trace 信息。
4. 可完成 KB 基础 CRUD。
5. 可查看渠道健康与网关状态摘要。

## 8.2 稳定性验收

后端必须通过：

1. `make validate-structure`
2. `make lint`
3. `make typecheck`
4. `make test`
5. `make check`
6. `make acceptance`

前端必须通过（如引入 Node 工程）：

1. 前端 lint
2. 前端 typecheck
3. 前端 unit/component tests
4. 前端 e2e（至少覆盖第 9.3 的关键场景）

## 8.3 可观测验收

1. 至少支持按 `trace_id/ticket_id/session_id` 任一维度检索链路。
2. 关键事件可见：`ingress_normalized`、`route_decision`、`sla_evaluated`、`handoff_decision`。
3. Ticket 动作事件在 timeline 中可回溯到 actor 与时间戳。

## 8.4 约束符合性验收

1. OpenClaw 未承载业务规则。
2. 未引入 autonomy-first 无约束循环代理。
3. 升级2实现未破坏 workflow/core 的业务主干。

## 9. 测试方案

## 9.1 后端测试

1. 单元测试：新增 API 服务层的参数校验、映射与错误处理测试。
2. 集成测试：覆盖页面依赖的 `/api/*` 端点主路径与失败路径。
3. 工作流/回归测试：保持现有 `tests/workflow`、`tests/regression` 全绿。
4. 验收回放：执行 `make acceptance`，生成 `storage/acceptance` 报告。

## 9.2 前端测试

1. 组件测试：Table/Filters/StatusBadge/ActionsPanel/Timeline。
2. 页面测试：Dashboard、Ticket List、Ticket Detail、Trace、Queue、KB、Channels。
3. 状态测试：Loading/Empty/Error、筛选持久化、分页与排序交互。

## 9.3 E2E 关键场景（至少）

1. 场景A：从 Ticket List 筛选进入 Ticket Detail，执行 claim 并看到状态变化。
2. 场景B：在 Ticket Detail 执行 escalate/resolve/close，Timeline 出现对应事件。
3. 场景C：从 Dashboard/Trace List 进入 Trace Detail，查看 route/tool/grounding 信息。
4. 场景D：KB 新增文档并在列表可见，再编辑、删除并验证结果。
5. 场景E：Channels 页面看到 connected/last_error/retry_state 与 gateway status。

## 9.4 验收证据要求

验收提交时必须附：

1. 测试命令输出摘要。
2. 关键 e2e 执行记录（日志或截图）。
3. `make acceptance` 结果摘要。
4. 未完成项与风险清单。

## 10. DoD（Definition of Done）

满足以下全部条件才可判定“升级2完成”：

1. 本文第 8 章所有 MUST 项均通过。
2. 本文第 9 章测试最小集执行通过并有证据。
3. 非目标约束未破坏。
4. 文档、代码、测试与演示材料已同步更新。
