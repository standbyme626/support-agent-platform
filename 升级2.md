我会基于当前仓库状态来设计，而不是凭空发散：

* README 已明确：项目是 **workflow-first、agent-assisted**，OpenClaw 只负责 `ingress/session/routing`，**当前阶段不包含前端后台**。([GitHub][1])
* Chatwoot 本身是开源自托管客服平台，适合借鉴 **客服工作台 / inbox / assignee / 多渠道 support** 的产品组织。([GitHub][2])
* Zammad 是 web-based 的开源 helpdesk/customer support system，支持多渠道客户沟通，适合借鉴 **ticket / queue / SLA / 协同处理** 的工作台结构。([GitHub][3])
* OpenClaw 官方也明确了它的强项在 **Gateway / sessions / routing / channel connections**，所以前端要围绕 ticket/case 运维，而不是把 OpenClaw 做成业务后台。([OpenClaw][4])

# 升级2：前端目录结构 + 页面级组件拆分 + API 对接清单

> 附属规范文档（必读）：[`升级2-实施要求与验收测试规范.md`](./升级2-实施要求与验收测试规范.md)
> 执行任务分解（A1~H4）：[`升级2-任务分解与执行清单-A1-H4.md`](./升级2-任务分解与执行清单-A1-H4.md)

## 1. 前端定位

这次前端**不是用户聊天页**，而是一个 **客服/运营工作台（Ops Console）**。

它的职责应该是：

* 看 ticket 列表
* 看 ticket 详情
* 看事件时间线
* 看 summary / recommended actions / similar cases
* 执行 claim / reassign / escalate / resolve / close
* 看 trace / SLA / handoff
* 管理 FAQ / SOP / 历史案例种子数据
* 看 OpenClaw / 渠道接入状态

这和 Chatwoot、Zammad 这类系统更接近，而不是普通 chat UI。([GitHub][2])

---

## 2. 前端技术建议

推荐直接做：

* **Next.js 15 + TypeScript**
* **Tailwind CSS**
* **shadcn/ui**
* **TanStack Query**
* **React Hook Form + Zod**
* **Recharts**
* **SSE 或轮询**

原因很简单：

* 你现在后端是 Python，前端独立出来最稳。
* 这个项目更需要“工作台 + 数据视图”，不是复杂拖拽交互。
* 后续你要做 trace、timeline、filters、tables，TanStack Query 和表格生态更合适。

* SLA 即将超时数量
* 已升级 ticket 数
* handoff 数量
* 渠道健康状态

* 队列负载排行

* `GET /api/queues/summary`
* `GET /api/channels/health`

## 4.2 Ticket List 页面

### 页面目标

这是整个工作台最常用的页面，类似 Chatwoot/Zammad 的主列表。([GitHub][2])

### 页面区块

* 全局搜索
* 多维筛选
* 表格列表
* 快捷操作
* 批量操作（第二阶段）

### 必须支持的筛选

* status
* priority
* queue
* assignee
* channel
* handoff_state
* service_type
* risk_level
* created_at range
* SLA state

### 组件拆分

* `ticket-filters.tsx`
* `ticket-status-badge.tsx`
* `ticket-priority-badge.tsx`
* `search-input.tsx`

* `GET /api/tickets`
* `GET /api/queues`

这是项目二最核心的“内部协同”落点。

**左栏：核心信息**

* 标题
* 状态
* 优先级

* assignee

**中栏：处理主视图**

* 对话消息
* summary
* recommended actions
* similar cases
* grounding source

**右栏：执行与状态**

* SLA
* handoff
* claim/reassign/escalate/resolve/close 操作

## 5.2 Ticket APIs

### `GET /api/tickets`

支持 query params：

* `page`
* `page_size`
* `status`
* `priority`
* `queue`
* `assignee`
* `channel`
* `service_type`
* `risk_level`
* `sla_state`
* `q`

### `GET /api/tickets/:ticketId`

返回 ticket 主体详情

### `GET /api/tickets/:ticketId/events`

返回事件时间线

### `GET /api/tickets/:ticketId/assist`

返回：

* summary
* recommended_actions
* risk_flags
* latest_messages
* provider
* prompt_version

### `GET /api/tickets/:ticketId/similar-cases`

返回相似历史工单列表

### `POST /api/tickets/:ticketId/claim`

body:

```json
{ "actor_id": "u_ops_01" }
```

### `POST /api/tickets/:ticketId/reassign`

body:

```json
{ "actor_id": "u_ops_01", "target_queue": "billing" }
```

### `POST /api/tickets/:ticketId/escalate`

body:

```json
{ "actor_id": "u_ops_01", "note": "需要主管审批" }
```

### `POST /api/tickets/:ticketId/resolve`

body:

```json
{ "actor_id": "u_ops_01", "resolution_note": "已处理完成" }
```

### `POST /api/tickets/:ticketId/close`

```json
{ "actor_id": "u_ops_01", "close_reason": "customer_confirmed" }
```

## 6. 公司定制化字段建议

你前面说要“加上公司需求内容，深度定制化”，那前端和 API 都要留这些字段：

* `service_type`
* `community_name`
* `building`
* `unit`
* `parking_lot`
* `parking_space_id`
* `rent_contract_id`
* `complaint_category`
* `visit_required`
* `follow_up_required`
* `approval_required`
* `risk_level`

这些字段不要先全做成复杂配置系统，先做到：

* 列表可筛选
* 详情可展示
* summary 可引用
* recommended actions 可结合

# 升级2 UI 草图说明（低保真布局）

## 1. 全局样式规范

**页面统一结构**

```
┌─────────────────────────────────────────────────────────┐
│ Top Navigation (logo + global search + user menu)       │
├─────────────────────────────────────────────────────────┤
│ Sidebar (Dashboard / Tickets / Queues / Traces / KB /… )│
├─────────────────────────────────────────────────────────┤
│ Breadcrumb (optional)                                    │
├─────────────────────────────────────────────────────────┤
│ Main Content                                            │
│                                                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**样式约定（Tailwind）**

| 语义       | Tailwind class 建议                                                  |
| -------- | ------------------------------------------------------------------ |
| 主背景      | `bg-gray-50`                                                       |
| 主卡片      | `bg-white shadow-sm rounded-lg p-4`                                |
| 页头       | `text-xl font-semibold text-gray-800`                              |
| 状态 Badge | `inline-flex items-center px-2 py-0.5 rounded text-sm font-medium` |
| 错误提示     | `text-red-600`                                                     |
| 空态提示     | `text-gray-500 italic`                                             |

统一配色（可品牌化）：

* 成功状态：`green-500/600`
* 警告状态：`yellow-500/600`
* 错误状态：`red-500/600`
* 默认 info：`blue-500/600`

## 2. Dashboard 首页

**目标**
整体运营态势览图，快速看 ticket 状态、SLA 风险、渠道健康、trace 异常。

**低保真布局**

```
┌─────────────────────────────────────────────────────────┐
│ Dashboard – 全局概览                                     │
├─────────────────────────────────────────────────────────┤
│ ◉ Overview Cards ─ statistics                            │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐          │
│  │ New    │  │ InProg │  │ SLA Wrn│  │ HandOff│          │
│  │ Tickets│  │ Tickets│  │ Tickets│  │ Pending │         │
│  └────────┘  └────────┘  └────────┘  └────────┘          │
├─────────────────────────────────────────────────────────┤
│ ◉ Queue Workload (横向条形)                               │
│  Quick view of queues with high load                     │
├─────────────────────────────────────────────────────────┤
│ ◉ Traces with Errors (表格)                               │
│  Timestamp | workflow | traceId | state | error summary  │
└─────────────────────────────────────────────────────────┘
```

**说明**

* Top 4 cards 使用不同颜色表示状态
* Queue Workload 用 Bar Chart，超过阈值时显警告
* Trace with error 可支持点击 drill-down

## 3. Ticket List 页面

## 4. Ticket Detail 页面

**目标**
工单详情 + 事件时间线 + 活动面板 + SLA/Trace/摘要

**低保真布局**

```
┌─────────────────────────────────────────────────────────┐
│ Ticket Detail Header                                      │
│  T-ID | Title | Status Badge | Queue | Priority Badge     │
├─────────────────────────────────────────────────────────┤
│ Detail Grid                                               │
│ ┌───────────┬───────────────────────────────┬─────────────┐│
│ │ Left      │ Middle                        │ Right       ││
│ │ Info Card │ Conversation + Summary        │ Action Card ││
│ │           │ + Recommended Actions         │ SLA Card    ││
│ │           │ + Similar Cases                │ Handoff Card││
│ └───────────┴───────────────────────────────┴─────────────┘│
├─────────────────────────────────────────────────────────┤
│ Event Timeline (横向/竖向)                                  │
└─────────────────────────────────────────────────────────┘
```

**区域说明**

* **Left Info**：基础信息 & 业务字段（community/building/parking 等）
* **Middle**：对话 & summary & recommended actions & similar cases
* **Right**：可操作按钮 & SLA / handoff 状态

# 每个页面的 UX 细节注意事项

### 1) 筛选栏必须 persistent

筛选条件应保留（QueryString / LocalStorage）

---

### 2) 列表页面要有 Loading & Empty & Error 显示

```
┌──────────────┐
│ Loading...   │
│ (Skeleton UI)│
└──────────────┘
```

---

### 3) 时间 & SLA 表示风格统一

| 状态 | Badge Color | Tailwind                        |
| -- | ----------- | ------------------------------- |
| 正常 | green-500   | `bg-green-100 text-green-700`   |
| 警告 | yellow-500  | `bg-yellow-100 text-yellow-700` |
| 违规 | red-500     | `bg-red-100 text-red-700`       |

---

### 4) 表格交互细节

* 可排序
* 可筛选
* 支持多页
* 支持导出（可后期实现）
