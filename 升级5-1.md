# 升级5-1实施方案（非前端）：企业微信群协同、认领与会话收口

> 版本：v0.1（2026-03-13）
> 适用范围：升级5在企业微信与Ops API侧的补强；不包含前端工作台改造
> 约束：`升级5.md` 只读，本文件为升级5的补充落地方案

---

## 0. 开工门（Go/No-Go）

### 0.1 必须确认

1. 继续遵循 `workflow-first, agent-assisted`，不改成 fully autonomous。
2. 高风险终态动作必须 HITL 或显式命令确认，不允许模型自动执行。
3. 工单生命周期与会话生命周期保持分离：`ticket_end != session_end`。
4. 兼容入口 `v1 /api/tickets/:ticketId/close` 保留但受控，不做模糊语义扩展。
5. 企业微信侧支持“显式命令 + 受控自然语言”双入口，但终态动作只认显式格式。

### 0.2 禁止项

1. 禁止继续把 `/close` 当万能动作。
2. 禁止通过 summary-inference 对终态动作做模糊推断。
3. 禁止 deep agent 直接落库执行 `resolve/customer_confirm/operator_close/session_end`。
4. 禁止无 trace 的状态变更入库。

---

## 1. 范围定义（本次只做非前端）

1. 企业微信双群协同流程（维修群 + 工单群）。
2. 运维认领、处理、完结的命令与受控自然语言映射。
3. 会话结束命令（用于单人测试与上下文收口）。
4. Ops API v2 接线与 v1 close 兼容受控策略。
5. trace/audit 字段落地与可观测输出。

---

## 2. 角色与场景模型

## 2.1 双群模型

1. 维修群（用户群）：提交问题、补充信息、确认是否已恢复、可主动结束会话。
2. 工单群（运维群）：机器人播报待处理/处理中/已结束工单，运维在此认领与执行动作。

## 2.2 单人双角色测试模型

1. 同一测试人可在维修群扮演“提问者”，在工单群扮演“处理者”。
2. 允许一人完成完整闭环，但动作语义必须显式，防止上下文误触发。
3. 必须提供快速会话收口口令，避免上下文长期污染后续测试。

---

## 3. 命令与语义规范（企业微信）

## 3.1 终态动作仅接受显式命令

1. 认领：`/claim TCK-xxxx`
2. 已解决：`/resolve TCK-xxxx`
3. 用户确认关闭（系统动作）：`customer_confirm`（由用户中文确认话术触发映射）
4. 运维强制关单：`/operator-close TCK-xxxx 原因`
5. 结束会话（系统动作）：`session_end`（由用户中文收口话术或运维显式命令触发）

说明：`[summary-inference]` 仅可用于“建议”，不可直接触发上述终态动作。

## 3.2 自然语言入口（受控）

1. 运维群允许自然语言触发“建议动作”，例如“建议我认领这个工单”。
2. 机器人返回标准化建议与可执行命令，不直接执行终态动作。
3. 只有用户再次发送显式命令时才真正落库执行。

## 3.3 用户群会话收口（并存模式）

1. 用户群支持并存：
   - 显式命令：`/end`、`/new`
   - 中文自然话术：如“结束当前对话”“我有一个新问题，重新开始”
2. 结束当前会话：
   - 命令：`/end`
   - 中文：`“这轮先到这里，结束当前对话”`
   - 系统动作：`session_end`
3. 开启新话题：
   - 命令：`/new`
   - 中文：`“我有一个新问题，重新开始”`
   - 系统动作：`new_issue`
4. 恢复某工单上下文：
   - 中文：`“继续看一下工单 TCK-xxxx”`
   - 工单群可用显式切换动作（由运维端触发）。

## 3.4 并存解析优先级与AI理解策略

1. 解析优先级（必须）：
   - 第一优先：显式命令 `/end`、`/new`
   - 第二优先：中文规则短语（结束/新问题/继续当前）
   - 第三优先：disambiguation（继续当前 / 新问题 / 待澄清）
2. 高风险动作仍遵循显式命令与审批，不因自然语言直接执行终态变更。
3. AI理解策略：
   - 当前主链路采用规则判定（intent + new_issue_detector）驱动是否开新问题。
   - LLM可作为“建议层”补充置信度与解释，不直接替代第一、二优先级规则。
4. 不确定场景（置信不足）必须先澄清，不直接切换会话。

---

## 4. 运维认领后的用户可见性

1. 运维群认领成功后，机器人同步回执到维修群。
2. 回执最小字段：`ticket_id`、`当前处理人`、`认领时间`、`当前状态`。
3. 状态变更（resolve/customer_confirm/operator_close）均需双群可见，但可按配置脱敏处理人信息。
4. 默认策略：用户可见“有人在处理 + 当前阶段”，不暴露内部审批细节。

---

## 5. API与接线约束（对齐升级5主链路）

## 5.1 v2 主入口

1. `POST /api/v2/tickets/:ticketId/resolve`
2. `POST /api/v2/tickets/:ticketId/customer-confirm`
3. `POST /api/v2/tickets/:ticketId/operator-close`
4. `POST /api/v2/sessions/:sessionId/end`
5. `POST /api/v2/intake/run`
6. `POST /api/v2/tickets/:ticketId/investigate`

## 5.2 v1 兼容入口（受控）

1. 保留：`POST /api/tickets/:ticketId/close`
2. 规则：仅做受控转发，不做语义猜测。
3. 要求：请求中必须有明确动作意图或兼容策略命中，否则返回可解释错误。

---

## 6. Trace与审计（必写）

每一次动作执行必须记录以下字段：

1. `operator`（操作者，企业微信用户标识）
2. `ticket_id`
3. `source_session_id`（来源会话）
4. `result`（成功/失败/拒绝 + reason）

建议补充字段：

1. `channel`（wecom_user_group/wecom_ops_group/api）
2. `command_text`（原始命令）
3. `request_id`（幂等与排障）
4. `advice_only`（是否仅建议）

---

## 7. Deep Agent边界（只建议，不执行）

1. Deep agent 可做：调查、归因、SOP 检索、回复草稿、动作建议。
2. Deep agent 不可做：直接执行终态动作、绕过审批、隐式结束会话。
3. 所有高风险动作必须由显式命令或 HITL 决策层执行。

---

## 8. 分阶段实施清单（S1-S4）

## S1：命令协议与路由开关落地

### 范围

1. 固定命令词与参数格式。
2. 在企业微信适配层区分“建议输出”和“执行入口”。

### 退出标准

1. `/claim /resolve /customer-confirm /operator-close /end-session` 语法校验可用。
2. 非显式终态语句不会触发落库执行。

---

## S2：双群回执与可见性

### 范围

1. 运维认领后，自动回执维修群。
2. 工单状态变更双群同步模板落地。

### 退出标准

1. 认领后用户群可见处理进度。
2. 回执字段完整，trace 与消息内容可关联。

---

## S3：会话收口能力

### 范围

1. 用户群并存输入支持：`/end` `/new` 与中文收口话术共同生效。
2. session 与 ticket 状态读写彻底分离。

### 退出标准

1. 单人双角色连续测试不再出现“旧上下文卡死”。
2. `session_end` 不会误改 ticket 生命周期。

---

## S4：v2接线收口与兼容策略固化

### 范围

1. v2 票据动作与 intake/investigate 入口全接通。
2. v1 close 兼容规则文档化并测试化。

### 退出标准

1. v2 全动作可调用并返回 trace。
2. v1 close 兼容但受控，不出现模糊自动推断。

## 9.2 企业微信实测脚本版（每步话术 + 预期回复）

### 场景A：建单 -> 认领 -> 已解决 -> 用户确认关闭

1. 维修群发送：`空调不制冷，办公室 3A，今天一直有异响。`
2. 预期机器人回复（维修群）：`已创建工单 TCK-xxxx，状态：待认领。已通知处理同学。`
3. 工单群发送：`/claim TCK-xxxx`
4. 预期机器人回复（工单群）：`认领成功：TCK-xxxx，当前处理人：xxx。`
5. 预期机器人同步（维修群）：`工单 TCK-xxxx 已由 xxx 接手处理。`
6. 工单群发送：`/resolve TCK-xxxx`
7. 预期机器人回复（维修群）：`工单 TCK-xxxx 已处理完成，请确认是否恢复正常。`
8. 维修群发送：`已经恢复了，可以结单。`
9. 预期机器人回复（维修群）：`收到确认，工单 TCK-xxxx 已关闭。`

### 场景B：运维强制关闭（用户未回复）

1. 工单群发送：`/operator-close TCK-xxxx 用户离线，电话确认恢复`
2. 预期机器人回复（工单群）：`已强制关闭 TCK-xxxx，原因已记录。`
3. 预期机器人同步（维修群）：`工单 TCK-xxxx 已由处理同学关闭，原因：用户离线，电话确认恢复。`

### 场景C：用户群结束会话（不使用命令）

1. 维修群发送：`这轮先到这里，结束当前对话。`
2. 预期机器人回复（维修群）：`好的，本次会话已结束。后续可随时发新问题。`
3. 维修群再次发送：`打印机现在又报错了。`
4. 预期机器人回复（维修群）：`已按新会话处理，并创建/关联新工单上下文。`

### 场景D：用户群开启新话题（不继承旧上下文）

1. 维修群发送：`我有一个新问题，重新开始：会议室投屏连不上。`
2. 预期机器人回复（维修群）：`已开启新问题处理流程，正在为你生成新的工单记录。`

### 场景D2：用户群显式命令结束与新建（并存验证）

1. 维修群发送：`/end`
2. 预期机器人回复（维修群）：`当前会话已结束，可继续发起新问题。`
3. 维修群发送：`/new`
4. 预期机器人回复（维修群）：`已切换到新问题模式，请描述你的新问题。`

### 场景E：运维群自然语言仅给建议

1. 工单群发送：`请帮我结束这个工单。`
2. 预期机器人回复（工单群）：`建议执行：/resolve TCK-xxxx 或 /operator-close TCK-xxxx 原因。`
3. 预期行为：机器人不直接执行终态动作，等待显式命令。

## 9.3 验收证据

1. 修改文件清单。
2. 测试结果摘要（通过/失败与原因）。
3. trace 样例（至少含 1 次 claim、1 次 resolve、1 次 end-session）。
4. 风险点与回滚策略。

---

## 10. 风险与缓解

1. 风险：自然语言误触发终态动作。
   缓解：终态动作仅显式命令执行，默认 advice-only。
2. 风险：单人测试导致身份混淆。
   缓解：trace 强制记录 `operator` 与 `channel`，并在回执展示角色来源。
3. 风险：v1 兼容入口语义漂移。
   缓解：将兼容策略固化为白名单映射并加回归测试。
4. 风险：会话与工单再次耦合。
   缓解：`session_end` 与 `ticket_close` 分开 API、分开事件、分开状态对象。

---

## 11. 结论

本方案用于补齐升级5在“企业微信协同与操作收口”侧的落地缺口，聚焦非前端范围。核心目标是：

1. 动作语义显式化。
2. 双群协同可见化。
3. 会话收口可操作。
4. trace 审计可复盘。
5. deep agent 保持建议角色，不越权执行。
