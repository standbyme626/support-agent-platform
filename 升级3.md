一、升级三定位
升级三不是继续补前端页面，而是在已完成的工单工作台基础上，把系统升级为一个 workflow-first、agent-assisted 的内部服务台 Agent 平台。
• 工作流负责主干：建单、分流、升级、关闭、SLA、handoff 等高约束流程由预定义工作流控制。
• Agent 负责增强：在关键节点提供分类、检索、摘要、推荐动作、运营问答和协同问答能力。
• OpenClaw 只负责入口层：ingress、session、routing、channel connections，不承担工单业务决策。
• 高风险动作必须可人工确认：审批前暂停，批准后恢复执行。
• 因此，升级三的目标不是做一个“更会聊天的机器人”，而是做一个具备真实模型能力、稳定检索能力、可靠接入能力、可审批可追踪能力的内部服务台 Agent 平台。
二、升级三总体目标
• 把 AI 能力从“可接入”升级到“可落地”：真实 provider、prompt version、trace metadata、fallback、错误路径。
• 把检索能力从“能用”升级到“好用”：FAQ / SOP / 历史案例的混合检索、向量检索、rerank 与 grounding。
• 把入口接入从“打通”升级到“可靠”：企业微信 / OpenClaw / webhook 的签名校验、幂等、防重放、失败重试、replay/debug。
• 把人机协同从“人工接手”升级到“可审批、可恢复”：高风险动作进入 HITL / 审批流。
三、升级三总体结构
• 一套共享核心：ticket / case core，session / routing core，retrieval / knowledge core，trace / audit core，workflow engine。
• 三个工作流：员工请求入口工作流、处理人员协同工作流、前端工作台处理工作流。
• 四个 Agent：Intake Agent、Case Copilot Agent、Operator / Supervisor Agent、Dispatch / Collaboration Agent。
• 四大增强模块：真实 LLM Provider 能力、检索增强、接入可靠性增强、Human-in-the-loop / 审批闸门。
四、三类工作流（升级三正式定义）
4.1 员工请求入口工作流
员工在企业微信中通过 @机器人 发起报修、咨询、投诉、人工服务请求和进度查询，系统自动完成意图识别、FAQ 直答、自动建单、handoff 请求、escalation 请求和进度查询回复。
• 典型输入：停车位抬杆坏了、我要人工服务、我要投诉、我的工单处理到哪了。
• 典型输出：FAQ 回复、工单编号、当前状态、handoff_requested、escalation_requested、当前工单进度说明。
• 状态流转：message_received → classified → faq_answered / ticket_created / handoff_requested / escalation_requested / progress_queried。
• 页面映射：Dashboard、Ticket List、Ticket Detail、Timeline。
• API 映射：POST /api/intake/message、GET /api/tickets/:ticketId、GET /api/tickets/:ticketId/events、GET /api/tickets/by-session/:sessionId（可选）。
• 升级三增强点：真实模型接入、FAQ/SOP/历史案例 grounding、投诉/人工服务识别、进度查询回复。
4.2 处理人员协同工作流
维修、客服和值班人员在内部处理群中接收新工单提醒、待认领提醒、SLA 风险提醒，并查询“我的工单”“最急工单”，支持轻量认领与查看摘要。
• 典型输入：有没有新工单、派给我的工单有哪些、哪些 SLA 快超时、认领 TCK-123、TCK-123 的摘要是什么。
• 典型输出：新工单列表、我的工单列表、待认领工单、最急工单、工单摘要、认领结果。
• 状态流转：pending_claim、claimed、waiting_human、sla_warning、sla_breached、escalated。
• 页面映射：Queue、Ticket List、Ticket Detail、Channels / Gateway。
• API 映射：GET /api/tickets?assignee=me、GET /api/tickets?status=pending_claim、GET /api/queues/summary、POST /api/tickets/:ticketId/claim、GET /api/tickets/:ticketId/assist。
• 升级三增强点：群内 query 语义识别、轻量 action、“最急工单”排序逻辑、新单/风险主动推送策略。
4.3 前端工作台处理工作流
客服、维修、运营、主管在 Web Console 中完成正式处理，包括工单认领、转派、升级、关闭、知识库维护以及运营监控。
• 典型输入：打开 Ticket List、查看 Ticket Detail、查看队列状态、查看 Trace、管理 FAQ/SOP/历史案例、认领/转派/升级/关闭。
• 典型输出：claim、reassign、escalate、resolve、close、知识库更新、运营判断结果。
• 状态流转：new、in_progress、claimed、waiting_human、reassigned、escalated、resolved、closed。
• 页面映射：Dashboard、Ticket List、Ticket Detail、Event Timeline、Trace List/Trace Detail、Queue、Knowledge Base、Channels/Gateway。
• API 映射：dashboard、tickets、events、assist、actions、traces、queues、channels、openclaw、kb 等核心接口。
• 升级三增强点：工单 Copilot、运营 Copilot、更稳的 summary/recommendations、更可解释的 trace 与 risk summary。
五、四个 Agent（升级三正式定义）
5.1 Intake Agent
负责员工请求入口的理解、分类与建单建议。
• 主要职责：意图分类、FAQ/SOP/历史案例检索、建单建议、初始 summary、进度查询回复。
• 工具：classify_intent、search_kb、search_history_cases、create_ticket、get_ticket_status。
• 边界：不直接关闭工单，不越权做高风险升级裁决。
• 升级三增强：接真实模型、provider trace、prompt version、更稳的 handoff / escalation 识别。
5.2 Case Copilot Agent
为单张工单提供 AI 辅助。
• 主要职责：summary、recommended actions、similar cases、risk flags、handoff / escalate suggestion。
• 工具：get_ticket、get_ticket_events、search_kb、search_similar_cases、get_sla_state。
• 边界：给建议，不直接执行敏感动作。
• 升级三增强：summary 稳定化、recommendations 更贴业务、similar cases 命中率提升、grounding 来源更清晰。
5.3 Operator / Supervisor Agent
回答运营和主管视角的问题。
• 主要职责：最紧急工单 Top N、最堵队列、需要人工接管的 case、SLA 风险集中点、trace 异常是否影响真实业务。
• 工具：list_tickets、get_queue_summary、list_sla_risks、list_handoffs、list_trace_errors。
• 边界：只分析与排序，不直接执行业务动作。
• 升级三增强：Dashboard Copilot、Queue Copilot、跨页运营问答能力。
5.4 Dispatch / Collaboration Agent
支持内部处理群协同。
• 主要职责：回答“我的工单”“最急工单”“待认领工单”，返回工单摘要，支持轻量认领。
• 工具：list_my_tickets、list_pending_claim、list_urgent_tickets、get_ticket_summary、claim_ticket。
• 边界：不在群里直接做高风险关闭 / 升级 / 改派，高风险动作跳前端工作台。
• 升级三增强：query 语义理解、主动推送策略、群内轻量 action 与前端一致性。
六、四大增强模块
6.1 模块 A：真实 LLM Provider 能力
• 目标：把现有 AI 层升级为真实、稳定、可追踪的模型能力。
• 主要建设内容：真实 provider、provider/model 配置外置、prompt version、trace metadata、fallback、summary/recommendations/similar cases 真走真实模型。
• 验收标准：至少 1 个真实 provider 跑通，Intake Agent 和 Case Copilot Agent 真走模型，错误路径与回退路径清晰，所有模型调用可追踪。
6.2 模块 B：检索增强（混合检索 / 向量库 / grounding）
• 目标：提升 FAQ / SOP / 历史案例检索质量。
• 主要建设内容：lexical + vector 混合检索、向量索引、rerank、source attribution、相似案例增强。
• 验收标准：三类知识源统一召回，similar cases 质量提升，AI 输出带 grounding 来源。
6.3 模块 C：接入可靠性增强
• 目标：把企业微信 / OpenClaw 接入做成可稳定运行的生产形态。
• 主要建设内容：webhook 签名校验、nonce/timestamp/replay 防护、幂等、失败重试、trace_id 贯穿 inbound/outbound、route/session 健康检查、replay/debug。
• 验收标准：webhook 重放不重复建单，inbound/outbound 全链路有 trace，gateway / channel health 与实际一致。
6.4 模块 D：Human-in-the-loop / 审批闸门
• 目标：把高风险动作纳入人工审批与恢复执行体系。
• 主要建设内容：至少 1~2 类高风险动作审批流、agent 执行暂停/恢复、handoff 保留上下文、前端展示 pending approvals / waiting human。
• 适用动作示例：投诉升级、费用争议、敏感工单关闭、高风险回复、改派关键责任人。
• 验收标准：高风险动作能暂停等待审批，审批后可恢复执行，handoff 前后上下文完整。

九、页面与升级三功能映射
9.1 Dashboard 总览页
• 升级目标：从“运营看板”升级成“运营 Copilot 工作台”。
• 新增能力：运营 Copilot 面板、今日最紧急工单 Top N、最堵队列、建议人工接管、trace 异常对业务影响、SLA 风险总结。
• 对应 Agent：Operator / Supervisor Agent。
• 对应 API：GET /api/dashboard/summary、GET /api/queues/summary、GET /api/traces、POST /api/copilot/operator/query（建议新增）。
9.2 Ticket List / Inbox
• 升级目标：从“工单列表页”升级成“可被 agent 驱动的任务入口页”。
• 新增能力：AI 推荐优先处理顺序、标注建议人工接管工单、更多定制字段筛选、按风险/hand-off/审批状态筛选。
• 对应 Agent：Case Copilot Agent，间接关联 Dispatch / Collaboration Agent。
• 对应 API：GET /api/tickets、风险/hand-off/审批状态查询，以及 POST /api/copilot/ticket-list/rank（建议新增）。
9.3 Ticket Detail
• 升级目标：把 Ticket Detail 做成真正的 Case Workspace。
• 新增能力：更强的 summary、更 grounded 的 recommended actions、similar cases、risk flags、需要审批的动作展示、handoff 上下文保留。
• 对应 Agent：Case Copilot Agent。
• 对应 API：ticket detail、events、assist、similar-cases、grounding-sources、approve-action（建议新增）。
9.4 Queue 页面
• 升级目标：从“队列统计页”升级成“队列决策页”。
• 新增能力：队列 Copilot 解释堵点、建议调度、识别 SLA 风险。
• 对应 Agent：Operator / Supervisor Agent。
• 对应 API：GET /api/queues/summary、GET /api/queues/workload、POST /api/copilot/queue/query（建议新增）。
9.5 Trace List / Trace Detail
• 升级目标：让 trace 成为“解释 AI 为什么这么做”的页面，而不是纯技术表。
• 新增能力：retrieved docs 与 grounding 更清晰、provider / model / latency / prompt version 展示、trace 与 ticket / session / workflow 联动、business impact 解释。
• 对应 API：GET /api/traces、GET /api/traces/:traceId、GET /api/traces/:traceId/business-impact（建议新增）。
9.6 Knowledge Base
• 升级目标：从“知识列表页”升级成“grounding 资产管理页”。
• 新增能力：FAQ/SOP/history cases 检索状态、命中率低的知识源、知识缺口、相似案例质量反馈。
• 对应 Agent：Intake Agent、Case Copilot Agent。
• 对应 API：KB CRUD + coverage / misses（建议新增）。
9.7 Channels / Gateway
• 升级目标：把渠道页做成“接入可靠性中控页”。
• 新增能力：签名校验状态、replay/duplicate 检测、最近失败重试、route/session 健康、inbound/outbound trace 串联。
• 对应 API：channels health/events、openclaw status/routes/replays/retries/sessions/signature-status（部分建议新增）。
十、状态机设计建议
• 工单业务状态：new、claimed、in_progress、reassigned、escalated、resolved、closed。
• 人机协同状态：handoff_requested、waiting_human、pending_claim、pending_approval。
• 系统辅助状态：classified、retrieved、summary_generated、recommendation_generated、trace_recorded。
十一、接口扩展建议
• AI Copilot 接口：POST /api/copilot/operator/query、POST /api/copilot/queue/query、POST /api/copilot/ticket/:ticketId/query、POST /api/copilot/dispatch/query。
• 检索增强接口：GET /api/tickets/:ticketId/grounding-sources、GET /api/tickets/:ticketId/similar-cases、POST /api/retrieval/search、GET /api/retrieval/health。
• 审批接口：GET /api/approvals/pending、POST /api/approvals/:approvalId/approve、POST /api/approvals/:approvalId/reject、GET /api/tickets/:ticketId/pending-actions。
• 渠道可靠性接口：GET /api/openclaw/retries、GET /api/openclaw/replays、GET /api/openclaw/sessions、GET /api/channels/signature-status。
十二、测试与评测建议
• LLM 评测：分类正确率、summary 质量、recommended actions 质量、handoff 建议准确性。
• 检索评测：FAQ 命中率、SOP 命中率、历史案例相似性质量、lexical-only vs hybrid 差异。
• 接入可靠性测试：webhook 重放、重试逻辑、幂等、session 恢复。
• HITL 测试：高风险动作是否暂停、审批后是否恢复、拒绝后是否安全终止、上下文是否完整保留。
十三、分阶段实施建议
• 阶段 A：模型层 —— provider、prompt version、trace metadata、fallback。
• 阶段 B：检索层 —— vector / hybrid retrieval、rerank、source attribution、similar cases 质量提升。
• 阶段 C：接入层 —— 签名校验、replay guard、retry manager、session / route 健康检查。
• 阶段 D：审批与协作层 —— pending approvals、HITL runtime、pending human actions、高风险动作恢复执行。
十四、升级三验收标准（总表）
• 模型层：至少 1 个真实 provider 跑通；summary / recommendations / similar cases 接真实模型；provider / latency / errors 可追踪。
• 检索层：FAQ / SOP / 历史案例统一召回；hybrid / vector 检索可用；grounding source 可展示；similar cases 质量提升可证明。
• 接入层：webhook 签名校验可用；replay / duplicate 不重复建单；retry 可观测；gateway / session / route 状态可查。
• 审批层：至少 2 类高风险动作有审批；审批前不会直接执行；审批后可恢复；handoff 保留上下文。
• 工作流 / Agent 层：3 个工作流边界清晰；4 个 agent 职责清晰；Dashboard 有 Operator Copilot；Ticket Detail 有 Case Copilot；内部处理群 workflow 跑通。
十五、不做项（范围约束）
• fully autonomous planner
• 多 agent 自由自治
• OpenClaw 业务化
• 重型 CRM / ITSM 套件化
• 大规模多租户 / RBAC
• 花哨但不落地的聊天前端
十六、A1～D4 任务清单（执行版）
A1. Provider 基础接入
• 定义 provider 抽象层：generate_text / generate_structured / generate_with_tools / healthcheck。
• 接入至少一个真实 provider（OpenAI-compatible），配置外置：OPENAI_BASE_URL / OPENAI_API_KEY / OPENAI_MODEL / LLM_TIMEOUT / LLM_RETRY。
• 让 Intake / Case Copilot / Operator / Dispatch 共用 provider 层，不再各自硬编码。
A2. Prompt Registry 与 Prompt Version
• 建立 intake、case copilot、operator、dispatch 四类 prompt registry。
• 每个 prompt 必须有 prompt_key、prompt_version、scenario、expected output schema。
• 支持版本切换与回滚，并在 trace 中显式记录 prompt_key + prompt_version。
A3. LLM Trace Metadata
• 记录 provider、model、prompt_key、prompt_version、latency、success/failure、request_id、token usage、retry count。
• 在 Trace Detail 和 Ticket Detail 的 AI Assist 区展示最小必要信息。
A4. Fallback 与错误路径
• 为 Intake / Case Copilot / Operator / Dispatch 四类 agent 设计 fallback。
• 模型不可用时主流程不崩：建单可继续、AI 可降级、前端有清晰提示。
B1. 知识源统一建模
• 统一 FAQ / SOP / History Cases 的 schema：doc_id、source_type、title、content、tags、updated_at、metadata。
• 建立 seed_data → normalized docs 的转换流程。
B2. 向量索引与 Hybrid Retrieval
• 保留 lexical retrieval；新增 vector retrieval 与 hybrid retrieval。
• 让 similar cases 可以用到新的 retrieval 层。
B3. Rerank 与 Grounding Source Attribution
• 增加 rerank 和 source attribution。
• Ticket Detail 展示 grounding sources；Trace Detail 展示 retrieved docs 与最终依据。
B4. 检索评测与知识缺口分析
• 构建 retrieval eval set，比较 lexical-only / hybrid / vector。
• 输出知识缺口：高频 query miss、重复工单缺 SOP、适合沉淀为 FAQ 的问题。
C1. 签名校验与来源校验
• 为 webhook 请求增加签名校验、timestamp / nonce 校验、非法请求拒绝与记录。
C2. Replay Guard 与幂等
• 建立 replay token / message fingerprint 和幂等键，防止重复建单与重复写事件。
C3. Retry 与 Outbound 稳定性
• 建立 retry manager，区分临时失败与永久失败；Channels 页面展示最近失败重试。
C4. Session / Route / Replay 调试能力
• 增强 gateway_status / trace_debug / replay_event 等工具与 Channels 页，支持 session→ticket 绑定与 replay 定位。
D1. 高风险动作识别规则
• 定义投诉升级、费用争议、敏感关闭、高风险回复、关键责任人改派等高风险动作，进入 pending_approval / waiting_human。
D2. Approval Runtime
• 建立 approval object、approve / reject / timeout 逻辑，并把审批事件写入 timeline / trace。
D3. 前端审批与人工接管视图
• 新增 Pending Approval 列表或面板；Ticket Detail 显示 pending actions；Dashboard 增加待审批卡片。
D4. Handoff 上下文保留与恢复执行
• 保存 handoff context（ticket、最近消息、summary、recommendations、grounding、trace）；批准后恢复，拒绝后安全回退；支持用户后续进度查询。
十七、Codex 执行总入口说明
• 建议按 A → B → C → D 顺序逐阶段推进，不要一次性全量并发。
• 每一阶段开始前，先检查现有代码路径和已有实现，不允许大规模推翻重写。
• 每完成一个子阶段，都必须输出：修改文件清单、新增文件清单、测试结果、剩余风险、是否可进入下一阶段。
• 任何阶段都要保持：workflow-first、agent-assisted、OpenClaw 只做 ingress / session / routing、高风险动作保留 HITL。
