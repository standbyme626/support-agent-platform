# 升级6存在问题记录

## 问题描述

### 问题1：并行处理双写问题

**现象**：同一消息同时触发了 procurement 系统和工单系统，两条处理链路并行执行。

**日志示例**：
```
智能工单机器人 3/20 14:31:49
已创建 procurement 记录 PR-2026-D85BDE39，状态：draft

智能工单机器人 3/20 14:31:52
工单 TCK-9D80964034 收到补充信息：申请采购2台显示器
```

**根因分析**：

1. **消息入口**：`scripts/wecom_bridge_server.py` 收到消息
2. **系统识别**：workflow 中的 `_resolve_system_for_outcome()` 正确识别出 `system=procurement`
3. **分发问题**：消息同时被发送给 procurement 系统和工单系统

**关键代码位置**：
- `workflows/support_intake_workflow.py:2237` - `_resolve_system_for_outcome()` 识别 system
- `workflows/support_intake_workflow.py:2517` - `_derive_ticket_action()` 决定 ticket_action
- `workflows/support_intake_workflow.py:596` - 两方法并行调用

**问题根源**：
- `_resolve_system_for_outcome()` 虽然正确识别了 system=procurement，但该返回值仅用于填充 `SupportIntakeResult.system` 字段
- `_derive_ticket_action()` 仍按原有工单逻辑判断，未考虑 system 是否为非工单系统
- wecom_bridge_server.py 的 `ticket_action` 分发逻辑未检查 system 字段，导致仍执行工单操作

**预期行为**：
- 识别为非工单系统（如 procurement）后，应跳过工单创建逻辑
- 仅执行目标系统的 create 操作
- 回执消息使用 `system:entity_id` 格式（如 `PR-2026-xxx`）

**修复方向**：
1. 在 `_derive_ticket_action()` 中，当 `system_key != "ticket"` 时，应返回非 create_ticket 的 action
2. 或在 wecom_bridge_server.py 的分发逻辑中，检查 system 字段，跳过工单操作
