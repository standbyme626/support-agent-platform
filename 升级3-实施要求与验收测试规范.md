# 升级3实施要求与验收测试规范

执行任务分解入口：[`升级3-任务分解与执行清单-A1-D4.md`](./升级3-任务分解与执行清单-A1-D4.md)

## 0. 升级3启动前必做 Checklist（开工门）

使用方式：升级3正式开始前逐项勾选。以下条目未全部完成时，不进入升级3联调与评测阶段。

### 0.1 边界与定位确认（必须）

- [ ] 已确认升级3目标是“可运营、可追踪、可审批、可恢复”的服务台能力升级，而非聊天机器人升级。
- [ ] 已确认 `workflow-first, agent-assisted` 主骨架不变。
- [ ] 已确认 OpenClaw 仍只承担 `ingress/session/routing`，不承载 ticket 业务规则。

### 0.2 升级1/2承接确认（必须）

- [ ] 已确认升级1交付的 ticket 生命周期、事件时间线、接入基础能力可复用。
- [ ] 已确认升级2交付的 Ops Console 页面与 `/api/*` 接口可承载升级3增强区。
- [ ] 已确认升级3不推翻升级2页面结构，只做“增强区”与“决策区”升级。

### 0.3 模型层准备（必须）

- [ ] 已冻结模型配置：`OPENAI_BASE_URL`、`OPENAI_API_KEY`、`OPENAI_MODEL`、`LLM_TIMEOUT`、`LLM_RETRY`。
- [ ] 已定义 Prompt Registry 字段：`prompt_key/prompt_version/scenario/expected_schema`。
- [ ] 已定义 LLM trace 字段：`provider/model/prompt_key/prompt_version/latency/request_id/token_usage/retry_count/success`。

### 0.4 检索层准备（必须）

- [ ] 已统一 FAQ/SOP/History Cases 文档 schema（含 `source_type` 与 `metadata`）。
- [ ] 已准备 retrieval eval set（至少 60 条，FAQ/SOP/历史案例均有覆盖）。
- [ ] 已定义检索指标口径：Top-K 命中率、grounding 覆盖率、similar-cases 可用率。

### 0.5 接入可靠性准备（必须）

- [ ] 已定义签名校验、timestamp/nonce、防重放规则。
- [ ] 已定义 replay token 与幂等键策略。
- [ ] 已定义 outbound retry 分级策略（临时失败/永久失败）。

### 0.6 审批与HITL准备（必须）

- [ ] 已定义至少 2 类高风险动作进入审批（例如投诉升级、费用争议、敏感关闭）。
- [ ] 已定义审批状态机：`pending_approval -> approved/rejected/timeout`。
- [ ] 已定义 handoff context 字段与恢复执行策略。

### 0.7 测试门禁准备（必须）

- [ ] 已确认后端门禁：`make validate-structure`、`make lint`、`make typecheck`、`make test`、`make check`、`make acceptance`。
- [ ] 已确认前端增强区测试策略（组件/页面/e2e）。
- [ ] 已确认回归范围：`tests/workflow`、`tests/regression` 全绿。

### 0.8 开工判定（Go/No-Go）

- [ ] Go：0.1~0.7 全部打勾。
- [ ] No-Go：任一关键项不满足，先补齐后开工。

## 1. 文档目的

本文件是 `升级3.md` 的执行约束与验收补充，定义三件事：

1. 升级3必须做到什么（可执行要求）。
2. 升级3做到什么才算通过（验收标准）。
3. 升级3如何测试并形成可复核证据（测试方案）。

适用范围：`support-agent-platform` 仓库内升级3相关后端、接入层、前端增强区、评测体系建设。

## 2. 基线与范围

### 2.1 当前基线（仓库事实）

1. 现有模型客户端：`llm/openai_compatible_client.py`、`llm/manager.py`。
2. 现有检索入口：`core/retriever.py`、`tools/search_kb.py`。
3. 现有接入链路：`openclaw_adapter/inbound_handler.py`、`openclaw_adapter/outbound_sender.py`、`openclaw_adapter/session_mapper.py`。
4. 现有工作台：`web_console/app/(dashboard)` 下 Dashboard/Tickets/Queues/Traces/KB/Channels。
5. 现有API承载：`scripts/ops_api_server.py` 与 `core/ticket_api.py`。

### 2.2 升级3目标范围

升级3范围固定为四大模块（A~D）：

1. A 模型层：真实 provider、prompt version、LLM trace、fallback。
2. B 检索层：hybrid/vector/rerank/source attribution、检索评测。
3. C 接入可靠性层：签名、防重放、幂等、重试、调试回放。
4. D 审批与HITL层：高风险动作审批、暂停恢复、上下文保留。

### 2.3 明确不做项（升级3范围外）

1. 不做 autonomy-first 的自由自治多Agent系统。
2. 不做 OpenClaw 业务化改造。
3. 不做重型 ITSM/CRM 套件化改造。
4. 不做大规模多租户/RBAC 重构。

## 3. 强制约束（MUST）

### 3.1 架构约束

1. `workflows/` 与 `core/` 继续承载业务主干。
2. OpenClaw 仅承载接入层职责。
3. 升级3不得破坏升级1/2主路径。

### 3.2 执行约束

1. Agent 默认“建议优先”，高风险动作不得直接执行。
2. 模型失败必须降级，不得阻断工单主流程。
3. 检索输出必须可解释，回答必须可溯源。

### 3.3 交付约束

1. 不允许无测试交付。
2. 不允许只改文档不落代码路径。
3. 不允许只改后端不验证工作台增强区。

## 4. 交付物要求

升级3完成时，至少应包含以下交付物：

1. 模型治理能力：provider 抽象、prompt registry、LLM trace、fallback。
2. 检索增强能力：vector/hybrid/rerank/source attribution。
3. 接入可靠性能力：signature/replay guard/idempotency/retry/debug。
4. 审批能力：approval runtime、pending actions、handoff context 恢复。
5. 工作台增强：Dashboard Copilot、Ticket Detail Copilot、Queue Copilot、Trace 解释区、Channels 可靠性区。
6. 自动化测试、评测报告、验收证据、风险清单。

## 5. 分阶段实施与退出标准

### 5.1 阶段A（模型层）

实现内容：

1. Provider 抽象与真实 provider 接入。
2. Prompt Registry 与版本记录。
3. LLM trace metadata 贯通。
4. Fallback 与降级路径。

退出标准：

1. Intake 与 Case Copilot 真实走模型。
2. Trace 可见 provider/model/version/latency/error。
3. 模型不可用时主流程可继续。

### 5.2 阶段B（检索层）

实现内容：

1. FAQ/SOP/History Cases 统一建模。
2. lexical + vector + hybrid 检索。
3. rerank 与 source attribution。
4. retrieval eval 与知识缺口分析。

退出标准：

1. 三类知识源统一召回。
2. Ticket Detail/Trace Detail 可见 grounding sources。
3. 有可复核检索对比报告。

### 5.3 阶段C（接入可靠性层）

实现内容：

1. webhook 签名与来源校验。
2. replay guard 与幂等键。
3. outbound retry 与失败分级。
4. gateway/session/route/replay 调试能力。

退出标准：

1. 重放消息不重复建单。
2. 重试路径可观测。
3. Channels 页面可见 reliability 指标。

### 5.4 阶段D（审批与HITL层）

实现内容：

1. 高风险动作识别策略。
2. approval runtime（approve/reject/timeout）。
3. 前端待审批视图与审批动作入口。
4. handoff context 保留与恢复执行。

退出标准：

1. 至少两类高风险动作进入审批。
2. 审批后可恢复，拒绝后安全回退。
3. timeline/trace 可完整回溯审批链路。

## 6. 功能与接口要求

## 6.1 模块能力要求（MUST）

### A 模型层

1. 必须支持统一 provider 调用接口。
2. 必须支持 prompt version 追踪与回滚。
3. 必须记录最小 LLM trace 字段并可查询。

### B 检索层

1. 必须支持 lexical/vector/hybrid 检索切换。
2. 必须支持 rerank 与 source attribution。
3. 必须支持 similar cases 与 grounding 联动输出。

### C 接入可靠性层

1. 必须支持签名校验、timestamp/nonce 校验。
2. 必须支持 replay/duplicate 防护与幂等。
3. 必须支持 retry 观测与 replay 调试。

### D 审批与HITL层

1. 必须支持 `pending_approval` 状态与审批动作。
2. 必须支持审批事件写入 timeline/trace。
3. 必须支持 handoff context 保留与恢复执行。

## 6.2 API分层要求（复用 / 增强 / 新增）

### 复用（保持兼容）

1. `GET /api/tickets`、`GET /api/tickets/:ticketId`、`GET /api/tickets/:ticketId/events`。
2. `GET /api/traces`、`GET /api/traces/:traceId`。
3. `GET /api/queues/summary`。

### 增强（升级3字段增强）

1. `GET /api/traces/:traceId` 增加模型与检索解释字段。
2. `GET /api/channels/health` 增加签名/重放/重试状态字段。
3. `GET /api/tickets/:ticketId/assist` 增加 grounding 与风险标记字段。

### 新增（升级3新增能力）

1. Copilot：
- `POST /api/copilot/operator/query`
- `POST /api/copilot/queue/query`
- `POST /api/copilot/ticket/:ticketId/query`
- `POST /api/copilot/dispatch/query`
2. Retrieval：
- `GET /api/tickets/:ticketId/grounding-sources`
- `POST /api/retrieval/search`
- `GET /api/retrieval/health`
3. Approval：
- `GET /api/approvals/pending`
- `POST /api/approvals/:approvalId/approve`
- `POST /api/approvals/:approvalId/reject`
- `GET /api/tickets/:ticketId/pending-actions`
4. Reliability：
- `GET /api/openclaw/retries`
- `GET /api/openclaw/replays`
- `GET /api/openclaw/sessions`
- `GET /api/channels/signature-status`

接口规范要求：

1. 列表接口支持 `page/page_size/total`。
2. 失败响应统一 `code/message/request_id`。
3. 动作接口写审计事件并可在 timeline 回溯。

## 6.3 关键字段要求

1. LLM trace 字段：`provider/model/prompt_key/prompt_version/latency/request_id/token_usage/retry_count/success`。
2. Grounding 字段：`source_type/source_id/title/snippet/score/rank/reason`。
3. Approval 字段：`approval_id/action_type/risk_level/status/requested_by/approved_by/rejected_by/timeout_at`。

字段落地最低标准：

1. 页面可展示。
2. API 可查询。
3. timeline/trace 可追踪。
4. summary/recommended actions 可引用。

## 7. 交互与体验要求

1. MUST：Copilot 回答必须展示来源与风险提示。
2. MUST：审批动作必须有确认弹窗与成功/失败反馈。
3. MUST：模型降级路径要给出明确提示，不可静默。
4. SHOULD：升级3增强区保持 Loading/Empty/Error 三态。
5. SHOULD：桌面与移动视口可正常加载，无结构性遮挡。

## 8. 验收标准

## 8.1 功能验收

1. A/B/C/D 四大模块全部联调可用。
2. 4个Agent（Intake/Case Copilot/Operator/Dispatch）至少各有1条主路径通过。
3. 高风险动作至少2类完成“暂停->审批->恢复/拒绝”闭环。
4. Ticket Detail 与 Trace Detail 可展示 grounding + LLM trace 信息。

## 8.2 稳定性验收

后端必须通过：

1. `make validate-structure`
2. `make lint`
3. `make typecheck`
4. `make test`
5. `make check`
6. `make acceptance`

前端（如改动 `web_console/`）必须通过：

1. 前端 lint
2. 前端 typecheck
3. 前端组件/页面测试
4. 前端 e2e（覆盖 9.4 关键场景）

## 8.3 量化验收阈值（升级3专属）

1. 模型调用成功率：主路径样本集不低于 95%。
2. 检索Top3命中率：hybrid 相对 lexical-only 提升不低于 15%。
3. Grounding覆盖率：Copilot输出带来源比例不低于 95%。
4. 防重放有效率：重复 webhook 不重复建单比例达到 100%。
5. 审批闭环成功率：高风险场景审批链路完成率达到 100%。

## 8.4 可观测验收

1. 至少支持按 `trace_id/ticket_id/session_id` 任一维度检索链路。
2. 可回溯关键事件：`ingress_normalized`、`route_decision`、`retrieved`、`summary_generated`、`approval_requested`、`approval_decision`。
3. 审批与动作事件可回溯 actor 与时间戳。

## 8.5 约束符合性验收

1. OpenClaw 未承载业务规则。
2. 未引入 autonomy-first 无约束循环代理。
3. 升级3未破坏升级1/2主链路。

## 9. 测试方案

## 9.1 后端测试

1. 单元测试：provider、prompt registry、retrieval、approval runtime、replay guard。
2. 集成测试：copilot/retrieval/approval/reliability 的 `/api/*` 端点主失败路径。
3. 回归测试：`tests/workflow`、`tests/regression` 全绿。
4. 验收回放：`make acceptance` 产出可复核报告。

## 9.2 前端测试

1. 组件测试：copilot panel、grounding card、pending approval list、approval dialog。
2. 页面测试：Dashboard、Ticket Detail、Queue、Trace、Channels 的升级3增强区。
3. 交互测试：审批确认、降级提示、错误反馈、筛选保留。

## 9.3 评测测试

1. LLM评测：分类准确率、summary质量、recommended actions质量、handoff建议准确性。
2. 检索评测：FAQ/SOP/history 命中率，lexical vs hybrid/vector 对比。
3. 接入评测：签名校验、防重放、幂等、重试。
4. HITL评测：审批暂停、批准恢复、拒绝回退、上下文完整性。

## 9.4 E2E关键场景（至少）

1. 场景A：员工消息进入 -> 模型分类 -> 生成summary -> trace可查。
2. 场景B：Ticket Detail 查看 grounding/similar cases -> 执行建议动作。
3. 场景C：同一 webhook 重放 -> 不重复建单。
4. 场景D：高风险动作进入审批 -> 审批通过后恢复执行。
5. 场景E：高风险动作审批拒绝 -> 安全回退并留审计轨迹。

## 9.5 验收证据要求

验收提交必须附：

1. 测试命令输出摘要。
2. 关键 e2e 场景日志或截图。
3. LLM评测与检索评测摘要。
4. `make acceptance` 报告摘要。
5. 未完成项与风险清单（blocking/non-blocking）。

## 10. DoD（Definition of Done）

满足以下全部条件才可判定“升级3完成”：

1. 第8章 MUST 项全部通过。
2. 第9章最小测试集执行通过并有证据。
3. 量化阈值达标或有可解释偏差说明。
4. 文档、代码、测试、评测、演示材料同步更新。
