1. 执行摘要
本计划书用于指导一个“support-agent-platform”从架构设计到分阶段落地。项目采取 workflow-first、agent-assisted 原则：系统主干使用可审计、可测试、可恢复的工作流；LLM/Agent 能力嵌入在分类、检索、摘要、推荐动作、转人工和协同节点中。
该系统不追求 AutoGPT/通用自主代理那种“无限自循环、自主规划一切”的形态，而是面向企业真实业务：客服受理、工单流转、内部协同、SLA 升级、人审与回溯。
OpenClaw 在本项目中只承担“消息入口 + Session + Routing”层，尤其适合把飞书、Telegram 等消息入口统一接入；企业微信则按‘自建应用/自定义 adapter → OpenClaw Gateway’路径纳入。
本计划书既给项目负责人阅读，也给 Codex 逐段执行。文档采用 A–Z 分组、A1/A2 等可点名任务粒度，配套验收标准、测试与回归要求。
2. 方法论：为什么选择 Workflow-First，而不是 Autonomy-First
通用 Agent 常被概括为 Task → Plan → Tool → Execute → Feedback 循环，这个抽象本身成立；但客服工单系统属于高约束、高审计、高人工接管场景。对于这类业务，系统主干更适合使用预定义工作流，Agent 能力只在局部节点增强。
•	应该固定的部分：是否建单、状态流转、SLA 规则、升级条件、审批条件、关闭条件。
•	适合 Agent 化的部分：意图分类、FAQ/SOP/历史工单检索、回复草稿、Case Summary、Recommended Actions、相似案例推荐。
•	不建议默认自治化的部分：开放式长期规划、无限循环、多轮自发拆任务、无约束工具探索。
因此，本项目的标准表述应为：Ticket-Centric Workflow System with Agent Capabilities（以 Ticket/Case 为中心、工作流优先、节点内引入 Agent 能力）。
3. 系统愿景与业务目标
•	让外部用户或一线客服在企业微信 / 飞书 / Telegram 等消息入口直接发起咨询。
•	让系统完成 FAQ 解答、工单创建、工单分类、优先级判断、转人工、内部协同、SLA 催办。
•	不优先建设前端后台；先把消息入口、后端核心与工单工作流打通。
•	系统必须支持后续增量扩展，但第一阶段聚焦客服/工单/helpdesk 场景。
两个项目壳子共享一套核心：项目一是对外客服工单受理，项目二是对内工单协同与升级。它们不是两套后端，而是同一套 Core Agent Runtime 上的两组 workflow。

6. OpenClaw 的定位、使用边界与配置原则
OpenClaw 很强，但本项目只使用它最值钱的部分：Gateway、channels、sessions、routing。不要把 OpenClaw 当业务引擎，也不要把 Ticket 决策逻辑塞进 OpenClaw。
•	要使用：Gateway、Feishu/Telegram 接入、Session、Bindings、Route back to original channel。
•	可以后续考虑：Broadcast groups、多个 agent workspace、WebChat。
•	暂不使用：Device nodes、远程节点本地执行、把工单规则写入 OpenClaw 内部。
6.1 推荐安装策略
建议安装 OpenClaw 官方当前完整发行版，但在配置上仅启用本项目需要的部分：一个 Gateway、少量 channels、bindings、日志与状态命令。策略是‘完整安装，最小使用’。
6.2 配置原则（面向 Codex）
原则 1：Model does not choose channel.
原则 2：Bindings are host-controlled and deterministic.
原则 3：OpenClaw owns session identity, but business system owns ticket identity.
原则 4：OpenClaw should never become the source of truth for ticket lifecycle.

7. 企业微信接入策略（重点说明）
本次核验未发现 OpenClaw 官方文档明确把企业微信（WeCom）列为官方内置标准渠道，因此本计划将企业微信视为‘企业微信自建应用 / 社区插件 / 自定义 adapter → OpenClaw Gateway’的接入路径。
•	推荐路径：企业微信自建应用接收消息回调 → 自定义 WeCom Adapter → OpenClaw Gateway → Core Agent Runtime。
•	如果后续社区插件足够稳定，可将 Adapter 替换为插件。
•	计划书与代码骨架都应确保：即使换渠道（企业微信 / 飞书 / Telegram），Core Agent Runtime 与 Ticket Store 基本不变。

11. 给 Codex 的强约束（必须遵守）
1.	不要实现一个通用聊天机器人。所有对话都必须能落到 ticket/case 对象，或与 ticket 关联的协同动作。
2.	不要把 OpenClaw 当业务引擎。它只负责 channel ingress、session 和 routing。
3.	不要优先做前端后台。MVP 先完成消息入口 + Ticket 后端 + 管理 API。
4.	不要只做 FAQ 问答。必须具备 create_ticket、update_ticket、assign/reassign、escalate、summary、handoff。
5.	必须保留 human handoff。复杂 case、补偿、投诉升级等要转人工，且保留上下文、证据和推荐动作。
6.	必须有测试。没有单元测试、接口测试、workflow 测试和回归测试，不能视为完成。
7.	必须把项目做成 workflow-first、agent-assisted，不得默认 autonomy-first。

13. 关键任务包细化示例（让 Codex 真的能按块执行）
D1 OpenClaw Gateway 接入
•	目标：完成官方当前版本安装、onboard、gateway status 与日志追踪脚本；只启用必要能力。
•	输入：OpenClaw 发行版、配置文件、绑定策略
•	输出：gateway 启动脚本、状态查询脚本、最小配置
•	测试：验证 Gateway 可启动/停止/重启，能记录日志
E3 企业微信 Adapter 骨架
•	目标：实现企业微信消息回调接入层，把消息标准化后送入 OpenClaw/内部入口。
•	输入：企业微信自建应用参数、回调密钥
•	输出：wecom_adapter 目录、回调 handler、签名校验
•	测试：模拟回调、签名验证、消息标准化测试
R3 自动建单
•	目标：在 intake workflow 中，当 FAQ 无法解决或满足建单条件时自动创建 ticket。
•	输入：标准化消息、分类结果、知识检索结果
•	输出：create_ticket 调用、ticket 记录、回写消息
•	测试：新 ticket 创建成功；重复 session 可复用同一 ticket/thread 关系
S3 /reassign
•	目标：在内部协同 workflow 中支持把工单从队列 A 转派到队列 B，并写入事件日志。
•	输入：ticket_id、target_queue、actor_id
•	输出：状态更新、event log、通知消息
•	测试：转派前后队列正确，日志完整，通知成功
Q3 SLA 自动催办与升级
•	目标：周期性检查 ticket 是否超时，并触发提醒或升级。
•	输入：sla_rules、ticket 时间戳
•	输出：提醒消息、escalation event、case-collab 推送
•	测试：超时 case 准确触发，未超时 case 不误报
14. 测试、回归与质量闸门
本计划书要求‘测试即交付的一部分’。每完成一组 A–Z 任务包，必须同步增加自动化测试，不允许实现与测试脱节。
测试层级	重点对象	必须覆盖的样例	通过标准
单元测试	Intent Router / Tools / Summary / Handoff	FAQ、投诉、费用争议、低置信度、转人工	核心函数边界与异常处理通过
集成测试	消息入口→Ticket→协同	Feishu/Telegram 入口、自动建单、claim/reassign/escalate	三条关键链路端到端通过
工作流测试	R/S 两条 workflow	FAQ 命中、创建工单、handoff、close、SLA 超时	状态机转移正确
回归测试	主路径稳定性	FAQ 回答、建单、summary、handoff、SLA	版本更新后结果一致或可解释变化
演示验收	可读性与稳定性	一线入口演示、内部协同演示、日志回放	项目负责人可顺利演示

•	必须建立种子数据与固定测试样本，避免 Codex 生成‘每次都变’的随机系统。
•	必须对 handoff 与 SLA 触发建立回归测试，因为这两处最容易被改坏。
•	必须为 session_id → ticket_id 映射建立测试，否则消息入口一变就容易丢上下文。

16. 为什么这个项目适合 2026 招聘市场
当前更有价值的不是‘完全放权给自治代理’，而是‘把 Agent 能力嵌进企业真实工作流’。本项目对外体现 AI ticketing / triage / handoff，对内体现 case collaboration / summary / SLA / recommended actions；这比泛化自治代理更贴近客服、工单、运营协同、内部流程自动化岗位。
•	优势不是“它会自循环”，而是“它能在高约束流程里可靠工作”。
•	优势不是“OpenClaw 用得多”，而是“知道 OpenClaw 只该负责接入与路由”。
•	优势不是“会聊天”，而是“能建单、能分流、能交接、能升级、能回溯”。
