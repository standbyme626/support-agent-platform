# 升级6实施规范（十系统全集成版）：可直接编码的接口与生命周期文档

> 版本：v0.2（2026-03-19）
> 适用范围：`support-agent-platform` 当前代码库（保留升级5边界）
> 原始材料：`升级6.backup.20260319.md`（保留，不覆盖）
> 目标：让 Codex 或开发同学可以“按文档直接实现”，避免接口与状态跑偏

---

## 0. 开工门（Go/No-Go）

### 0.1 必须确认

1. 架构仍是 `workflow-first, agent-assisted`，不是 fully autonomous。
2. OpenClaw 仍只承担入口/会话/路由，不承载业务规则。
3. 高风险动作（终态关闭、财务付款、审批终审）必须保留 HITL 与审计链路。
4. 升级6是“十系统集成样板”，不是十套完整 ERP/ITSM 重做。
5. 工单系统（L1）已存在，升级6以“接入统一协议”为主，不重写主线。
6. 十系统必须统一：接口风格、返回格式、错误格式、状态命名、ID 规则。

### 0.2 禁止项

1. 禁止把新业务规则继续堆进 `scripts/ops_api_server.py`。
2. 禁止十个系统各自定义不同返回壳和状态字段名。
3. 禁止跳过统一契约先做前端大改。
4. 禁止绕过 `trace_id/request_id/operator` 的审计字段直接改终态。

## 1. 方案定位与边界

### 1.1 目标

1. 在当前项目落地 10 个可演示的业务生命周期模块。
2. 建立统一 AI 协同层：意图识别 -> 字段抽取 -> 系统路由 -> 统一结果回传。
3. 提供完整目录与文件清单，做到“按清单能直接建项目骨架”。
4. 提供每系统 API 细化契约：路径、入参、出参、状态流转、错误码。

### 1.2 边界

1. 不替换现有工单核心能力，仅抽象为 L1 统一协议适配。
2. 首版允许 mock 数据与 JSON 仓储，不要求一次性上完整数据库迁移。
3. 每系统先做“1 条完整生命周期 + 3~8 核心接口 + 2~4 企微问法”。

## 2. 总体架构与请求流

### 2.1 主链路

企业微信一句话 -> AI 协同层 -> 意图识别 -> 字段抽取 -> 路由系统接口 -> 生命周期推进 -> 统一结果回传 -> 企业微信回复

### 2.2 AI 协同层固定 6 步

1. 接收标准化消息（含 `session_id`, `operator_id`, `channel`）。
2. 意图识别（命中 L1-L10 系统之一）。
3. 字段抽取（数量、预算、人员、日期、单号等）。
4. 知识优先（可直接回答时不落业务流）。
5. 调用目标系统 API（create/get/list/action/complete）。
6. 回写统一结果并记录 trace。

### 2.3 WeCom 群协作模式（当前执行版）

1. 工单系统（L1）采用双群模式：故障上报群 + 工单处理群。
2. 其余九个系统（L2-L10）采用单群模式：提问与处理在同一系统群完成。
3. 十个系统统一使用同一个机器人入口（智能工单机器人），不新增机器人实例。
4. 单群模式下必须使用命令化动作与固定回执模板，避免群聊噪音导致流程失控。
5. 分发映射优先级建议：`system` -> `queue` -> `inbox` -> `default`。
6. 当系统未命中显式映射时，回退到 `default`（当前默认工单处理群）。
7. WeCom 路由键建议按 `system:<system_key>`、`queue:<queue>`、`inbox:<inbox>` 统一命名，并在 trace 中保留命中键。
8. `user_receipt` / `collab_dispatch` 回执 metadata 必须带 `system` 与 `dispatch_matched_key`，用于审计与回放。

## 3. 统一接口与返回协议（全系统强制）

## 3.1 HTTP 约定

1. 协议：HTTP JSON。
2. URL 风格：资源名词为主，动作使用子资源。
3. 创建：`POST /api/<resource>`。
4. 查询：`GET /api/<resource>/{id}`。
5. 列表：`GET /api/<resource>?...`。
6. 动作：`POST /api/<resource>/{id}/<action>`。

## 3.2 请求头约定

1. `X-Request-Id`：外层请求唯一 ID（必填）。
2. `X-Trace-Id`：链路追踪 ID（必填）。
3. `X-Operator-Id`：操作人 ID（必填）。
4. `X-Idempotency-Key`：创建/终态动作建议必填。

## 3.3 成功返回壳（系统内部标准）

```json
{
  "ok": true,
  "system": "procurement",
  "entity_type": "request",
  "entity_id": "PR-2026-0001",
  "status": "pending_approval",
```

## 3.6 字段与命名规范

1. 状态字段统一为 `status`。
2. 时间字段统一为 `created_at`/`updated_at`（UTC ISO8601）。
3. 通用引用统一为 `entity_id`，系统内主键保留语义（如 `ticket_id`）。
4. 命名统一 `snake_case`。
5. 状态值统一小写下划线（如 `pending_approval`, `in_progress`, `closed`）。

## 3.7 ID 规则（前缀强制）

1. `T-YYYY-NNNN`（ticket）
2. `PR-YYYY-NNNN`（procurement_request）
3. `INV-YYYY-NNNN`（finance_invoice）
4. `AP-YYYY-NNNN`（approval_request）
5. `ONB-YYYY-NNNN`（hr_onboarding）
6. `ASSET-YYYY-NNNN`（asset）
7. `KB-YYYY-NNNN`（knowledge_article）
8. `CASE-YYYY-NNNN`（crm_case）
9. `PROJ-YYYY-NNNN`（project）
10. `GR-YYYY-NNNN`（goods_receipt）

## 3.8 HTTP 状态码约定

1. `200`：查询/动作成功。
2. `201`：创建成功。
3. `202`：异步动作已接收。
4. `400`：参数错误。
5. `404`：对象不存在。
6. `409`：状态冲突（非法状态迁移）。
7. `422`：语义校验失败。
8. `429`：限流。
9. `500`：系统错误。

### 动作执行统一响应要求

1. 动作成功：返回 `ok=true` 与动作后的 `status`。
2. 动作失败：返回 `ok=false`，`error.code` 必须使用第6.3节标准错误码。
3. 每次动作必须写入对应 `events/logs` 表，事件至少包含：`action`, `from_status`, `to_status`, `operator_id`, `trace_id`, `created_at`。
