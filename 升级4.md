# 升级4（优化版）

> 版本：v0.4.0-draft（2026-03-13）  
> 目标：在不推翻升级3的前提下，把“会话-工单关系、用户侧回复体验、多问题管理、人工接管体验”做成可落地产品能力。  
> 承接文档：`UPGRADE3_FINAL.md`、`README.md`、`ARCHITECTURE.md`

---

## 0. 为什么重写这版

原 `升级4.md` 的问题不是方向错，而是：

1. 内容重复（后半段存在大段重复拷贝）。
2. 建议与当前代码事实混写（已实现/未实现没有区分）。
3. “侃侃而谈”的价值点很多，但没有落到接口、数据、测试与阶段门。

这版优化保留核心观点，改为“**事实基线 -> 差距分析 -> 最小增量改造 -> 验收闭环**”。

## 1. 当前项目事实基线（代码核验）

以下为升级4设计必须承认的现状：

### 1.1 架构边界（已成立）

1. 仍是 `workflow-first, agent-assisted`，不是 autonomy-first。
2. OpenClaw 仅负责 ingress/session/routing，不承载 ticket 生命周期规则。
3. Ticket 生命周期规则在 `core/` + `workflows/`。

### 1.2 会话与工单关系（当前状态）

1. `SessionMapper` 目前是 `session_id -> ticket_id` 单值绑定模型。
2. `InboundHandler` 会把绑定中的 `ticket_id` 透传到 inbound metadata。
3. `wecom_bridge_server` 用 metadata 里的 `ticket_id` 作为 `existing_ticket_id`。
4. `WorkflowEngine` 在 `existing_ticket_id is None` 时创建新单，否则更新同一单。

结论：当前系统已具备“活跃工单”雏形，但还没有“同 session 多 ticket + 显式 active/recent 切换”模型。

### 1.3 用户侧回复能力（当前状态）

1. `ReplyGenerator` 已是 LLM-first + fallback。
2. 已有 prompt：`intake_user_reply / faq_reply / progress_reply / handoff_reply`。
3. 已有 reply trace 元数据（provider/model/prompt/fallback/degrade）。

结论：升级4不应重做回复体系，而应补“编排层 + 场景覆盖 + 可视化透出”。

### 1.4 Copilot能力覆盖（当前状态）

1. `ticket` copilot走 summary/retrieval链路。
2. `operator/queue/dispatch` 端点目前是规则化答案 + grounding，`llm_trace.provider=fallback`。
3. 路由已稳定：
   - `POST /api/copilot/operator/query`
   - `POST /api/copilot/queue/query`
   - `POST /api/copilot/ticket/:ticketId/query`
   - `POST /api/copilot/dispatch/query`

### 1.5 API与前端现状（与升级4直接相关）

已存在：

1. `GET /api/openclaw/sessions`（会话列表）
2. `GET /api/dashboard/summary`（6个运营指标）
3. Ticket Detail 已有“AI摘要 + 时间线 + grounding + 推荐动作”三栏布局

尚不存在：

1. `/api/sessions/:sessionId/*` 会话业务接口
2. `/api/copilot/disambiguate`
3. `/api/tickets/:ticketId/duplicates`
4. `/api/tickets/:ticketId/reply-trace`（当前通过 trace detail 间接查看）
5. 人工在工作台直接“编辑并发送用户回复”的显式 API/UI 闭环

## 2. 升级4真正要解决的问题

### 2.1 结构性问题（必须解决）

1. 同一 session 只能隐式绑定一个 ticket，导致多问题场景易混淆。
2. “继续当前问题”与“新问题”缺少标准判定流。
3. 咨询类消息与服务类消息分流粒度不够产品化。
4. 重复问题没有正式 duplicate/merge 建议机制。

### 2.2 体验性问题（后半段观点，必须保留）

1. 人工接管后，默认应是 `summary-first`，而不是全量上下文轰炸。
2. 详情应 `detail-on-demand`，可追溯而非默认展开。
3. 人工回复应是“AI草稿可编辑 -> 人工确认发送”，而不是裸手写，也不是全自动发送。
4. 用户侧应维持同通道连续性（不跳出会话体验）。

这部分不是“空谈”，是升级4的产品差异点。

---

## 3. 升级4目标与边界

## 3.1 四个总目标

1. 会话-工单关系正式升级为“1个active + N个recent”的显式模型。
2. 用户侧回复从“能力存在”升级为“编排可控、可解释、可观测”。
3. 同会话多问题处理（新问题识别/澄清/切换）闭环。
4. 人工接管体验落地为“摘要优先 + 详情按需 + 草稿可编辑发送”。

## 3.2 约束（必须）

1. 不破坏 `workflow-first` 主干。
2. 不把 OpenClaw 业务化。
3. 高风险动作继续走审批/HITL。

## 3.3 不做项（本期）

1. fully autonomous planner
2. 多租户/RBAC大改
3. CRM/ITSM 大而全复刻
4. 存储引擎重构

## 4. 升级4方案蓝图

## 4.1 三工作流 + 四Agent 协同场景（升级4必须保留）

### 4.1.1 固定骨架（不变）

1. 三工作流不变：
   - 员工请求入口工作流（Support Intake）
   - 处理人员协同工作流（Case Collaboration）
   - 前端工作台处理工作流（Ops Console）
2. 四Agent不变：
   - Intake Agent
   - Case Copilot Agent
   - Operator / Supervisor Agent
   - Dispatch / Collaboration Agent
3. 升级4做“职责增强”，不是“推翻重建”。

### 4.1.2 协同矩阵（谁主责、谁协同）

| 工作流 | 主责Agent | 协同Agent | 典型输入 | 关键决策 | 主要输出 |
| --- | --- | --- | --- | --- | --- |
| Workflow 1 员工请求入口 | Intake Agent | Case Copilot Agent | 员工消息（咨询/报修/追问） | 咨询 vs 服务、继续当前 vs 新问题、是否clarification | 直答/建单/切单/澄清问题/接管决策 |
| Workflow 2 协同处理 | Dispatch / Collaboration Agent | Operator Agent、Case Copilot Agent | `/claim /reassign /escalate /resolve /close` 与调度问答 | 分派优先级、是否触发审批、是否并单建议 | 状态流转、协同事件、调度建议、审批挂起/恢复 |
| Workflow 3 前端工作台 | Case Copilot Agent | Operator Agent、Dispatch Agent | 页面动作/API调用（查看、处理、审批、知识维护） | 摘要优先展示、人工回复编辑发送、风险动作审批 | AI摘要卡、推荐动作、可执行回复草稿、可审计处理记录 |

### 4.1.3 四Agent在升级4中的增强职责

1. Intake Agent：
   - 新增“新问题识别 + 歧义澄清”能力。
   - 对同会话多ticket提供入口判定。
2. Case Copilot Agent：
   - 从“单工单摘要”增强到“支持active/recent ticket上下文”。
   - 为人工接管提供更稳定的摘要与回复草稿。
3. Operator / Supervisor Agent：
   - 增加体验类指标解释（clarification触发率、咨询转工单率等）。
   - 支持运营视角下的误判诊断与阈值调优建议。
4. Dispatch / Collaboration Agent：
   - 增强多ticket协同与调度建议。
   - 继续承接协同命令链路与审批边界。

## A. 会话-工单编排层（Session-Ticket Orchestration）

### A1 目标

把“隐式单绑定”升级为“显式会话上下文”。

### A2 最小数据模型

新增 `SessionContext`（可存 `session_bindings.metadata`，先不新建大表）：

1. `session_id`
2. `active_ticket_id`
3. `recent_ticket_ids`（最多N条）
4. `session_mode`
5. `last_intent`
6. `updated_at`

### A3 最小接口

1. `GET /api/sessions/:sessionId`
2. `GET /api/sessions/:sessionId/tickets`
3. `POST /api/sessions/:sessionId/reset`
4. `POST /api/sessions/:sessionId/new-issue`
5. `POST /api/tickets/:ticketId/switch-active`

### A4 判定规则（首版）

1. 明确 `ticket_id` 提及 -> 指向该票
2. `progress_query` 且存在 active -> 指向 active
3. 意图突变 + 主题词差异高 -> `new_issue_detected`
4. 不确定 -> `awaiting_disambiguation`

### B1 目标

1. 明确 reply types：`greeting/faq/progress/handoff/generic/disambiguation/switch`

## 7. 验收指标（必须量化）

1. 会话多工单可用率：>= 95%
2. 新旧问题判定准确率（抽样）：>= 85%
3. clarification 误触发率：<= 15%
4. 咨询类直接答复率（不建单）：较升级前提升
5. 重复单建议命中率（抽样）：>= 70%
6. 用户侧 reply trace 可见率：>= 95%
7. 人工接管“摘要可用”满意度（内部评审）：>= 4/5

---

## 8. 风险与兜底

1. 风险：多工单判定误伤
   兜底：进入 clarification，不强判。
2. 风险：LLM不稳定导致体验波动
   兜底：保留 fallback + degrade_reason 观测。
3. 风险：并单建议误判
   兜底：只建议不强制，人工确认后执行。
4. 风险：人工接管后责任混乱
   兜底：发送动作强制记录 actor_id 与 ticket 状态变迁。

## 9. 本版升级4结论

升级4不是“加更多Agent”，而是把已有能力做成真正可用的产品逻辑层：

1. 会话和工单从“隐式绑定”升级为“可解释上下文”。
2. 用户回复从“有能力”升级为“有编排、有追溯”。
3. 同会话多问题与人工接管体验成为一等能力。
4. 后半段那些“看似在聊产品”的观点，已经被收敛为可实现的接口、页面和测试任务。

## 11. 附录B：关键思考保留（从原始升级4提炼）

> 下面这些不是“闲聊”，是升级4设计的核心判断。每条都标注来源类型。

### B1 `source-verified` 的事实判断

1. 项目主骨架必须保持 `workflow-first, agent-assisted`，不能变成 autonomy-first。
2. OpenClaw 边界仍应是 ingress/session/routing，不承载 ticket 业务决策。
3. 当前存在“session单ticket绑定”的现实约束，多问题场景会受影响。
4. 用户侧回复链路已经有 LLM-first + fallback，不应推翻重做。

### B2 `summary-inference` 的产品判断

1. “摘要优先、详情按需”比默认展开全量上下文更适合客服工作台。
2. 人工接管后最佳模式是“AI草稿 + 人工可编辑 + 人工发送”，而不是纯人工裸聊或全自动发送。
3. 同会话处理多问题是服务台成熟度分水岭，不只是体验优化。
4. 咨询类优先知识问答可降低无效建单，释放人工产能。

### B3 为什么保留这些思考

1. 它们直接决定前端信息密度与客服认知负担。
2. 它们直接决定“AI是否可感知”，而不只是后台有模型调用。
3. 它们直接决定升级4是否真的改善业务，而不是只改接口名。

## 13. 附录D：仍需决策的问题（实施前确认）

升级4完成后的预期（完整版本）

1. 用户侧（员工）预期

员工在同一会话内可以自然地“咨询 + 报修 + 查进度 + 再提新问题”，系统不再把所有消息硬塞进同一张单。
员工看到的是 AI 持续在线的自然回复，不是明显模板机。
咨询类问题（如“流程怎么走”“你能做什么”）优先知识问答，不会动不动建单或转人工。
服务类问题（报修、投诉、明确请求处理）会进入工单流，状态可追踪。
模糊场景下系统会主动澄清（继续当前问题/新建问题/切到旧单），减少误判。
2. 客服侧（工作台）预期

工单详情默认是 summary-first：先看 AI 摘要、当前状态、推荐动作、建议回复草稿。
详情是 detail-on-demand：点开再看原始消息流、grounding、相似案例、timeline、trace。
客服回复模式变成“AI草稿 -> 人工可编辑 -> 人工确认发送”，而不是纯手工，也不是全自动乱发。
人工接管后，仍保持同会话通道连续性，用户感知不中断。
客服可以明确处理“同一用户多个工单”的关系，不再靠记忆和手工比对。
3. 业务流程预期

会话与工单从“隐式单绑定”升级为“显式上下文”：
有 active_ticket
有 recent_tickets
有 session_mode
同会话多问题处理形成闭环：
新问题检测
歧义澄清
工单切换
再追问不误建单
重复问题治理形成闭环：
duplicate candidate
merge suggestion
人工 accept/reject
全链路可追溯
咨询优先与服务处理并存，不再二选一。
4. 技术与架构预期

不改变主架构：仍是 workflow-first, agent-assisted，不走 autonomy-first。
OpenClaw 边界不被破坏：仍只做 ingress/session/routing。
用户回复链路继续 LLM-first + fallback，但从“能力存在”升级到“编排清晰、可观测”。
Copilot能力将分阶段增强：先保持兼容，再把规则化回答逐步切到真实 LLM（尤其 operator/queue/dispatch）。
API层会出现会话业务接口、澄清接口、重复单建议接口，前端与后端的契约更明确。
5. 可观测性与治理预期

能按 ticket/session 看 reply 事件与关键元数据，定位“这句回复为什么这样生成”。
fallback/degrade 不再是黑盒，问题可诊断、可统计。
关键动作可审计：谁接手、谁编辑、谁发送、何时发送。
产品决策可数据化，不再只靠主观反馈。
6. 量化效果预期（验收口径）

会话多工单场景可用率达到稳定线（文档目标 >= 95%）。
新旧问题判定有可衡量准确率（文档目标 >= 85%）。
clarification 不会过度触发（文档目标 <= 15%）。
咨询类直接答复率提升，减少无效建单。
重复问题建议命中率可评估（文档目标 >= 70%）。
客服对“摘要可用性”的主观评分稳定提升（文档目标 >= 4/5）。
7. 业务层面的最终变化（你最关心的）

系统从“能跑”变成“真能用”。
员工感知到 AI 在持续协助，而不是偶发回复。
客服从“翻上下文 + 手动整理”变成“看摘要 + 修草稿 + 快速响应”。
管理层可以看到升级价值是否真实发生，而不是只看到功能上线。
8. 升级4完成后的系统定位

不是更大的 Agent 堆叠。
而是把“会话-工单-知识问答-人工接管”做成产品级协同闭环。
最终形态是：同会话多问题可控、AI与人工协作顺滑、体验和治理同时提升。
